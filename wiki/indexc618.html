  • Some information about included and dropped product items is available<ref name="testmatrix"> FP5 Cat Test Environment Matrix</ref>
  • SCM is working <ref name="svn">Flexman: FP5 Cat compilation environment</ref> and flexman has provided instructions<ref name="scm2">Flexman: Issues on WR SDK distribution</ref><ref name="svnfaq">Flexman: Flexi SVN FAQ</ref><ref name="ci">Flexman: FS5 Cat Continuous Integration builds</ref>.   • Platform Verification has announced that tey will use TestRMEX, TestLISA, TestAlarm and TestCPUMEM in Bobcat. The applications that will be used need to be migrated to the new run time environment (RTE).
  • Integration testing will use some applications.
  • TFRM traffic generators that run in regular Linux boxes cannot be used for performace measurement of routing products because of limited performance.
  • PV and SyVe are not going to test IR0<ref name="masterplan"/>, so it is possible to do most TA changes required by IR0 after IR0 is available.

[edit] General

  • Cat does not support upgrades from older FP releases to Cat<ref name="masterplan"/>. This means the test application, that are in principle mock user applications, might have to be re-written to some extent.
  • Cat does not include IPD nodes nor load balancers <ref name="testconfigurations">Test Configurations for FP5-Cat</ref>.
  • Cat will run on completely different hardware platforms than previous releases<ref name="masterplan"/><ref name="testconfigurations"/><ref name="referenceconfigurations">FP5-Cat reference configurations</ref>

[edit] Changes in FP that Affect Test Applications

[edit] Internal Release 0

  • FTR 2008-03-14
  • 4 new product items<ref name="releaseplan"/>
  • PR_2940 Support for Octeon Rack mount HW
    • SAT slides
    • Each physical Octeon chip will contain many FP nodes with different architectures. There can be one Linux node and many Simple Executive nodes.
    • No HAS in SE nodes.
    • Exsisting applications are not suitable for SE nodes, and converting them is not feasible.
    • New applications: We will be able to provide e.g. echo serves running in SE environment if needed for routing and performance tests. This has not been requested (yet). Software teams writing software for SE could be required to write test application also.
    • Data collector script in SS_TargetInfo needs minor changes that can be done when we have the environment available
    • Java will not be available in Octeon HW. This will make Java test applications obsolete: TestHASJavaAPI in SS_TestHAS in octeon enviroment.
  • PR_2944 Runtime environment redesign
    • FP5 RTE Overview
    • FP5 RTE Specification
    • Affects All applications
    • All RTE changes will be implemented in project J <ref>Project J Project Plan</ref>
    • RTE Changes are "partially backwards compatible". Complete rewrite of all subsystems is not needed.
    • Old paths, like /opt/Nokia, will stay as symlinks to new corresponding locations
    • Configuration should be stored in LDAP. LDAP contents can be presented as files to application. Editable files can be stored in global state volume and must be listed in /etc/gstatetab
    • Logs must go to /var/log/ not to /etc/opt/Nokia/log
    • $CLUSTERID variable will be provided
  • PR_2946: Fast intra-cluster messaging (FICM)
    • PDP Slides
    • No direct effects to test existing applications
    • New test application "maybe" requested for this feature.
    • Can be useful for TA instances that need to communicate with each other between nodes.
  • PR_3041 Embedded Operating System
    • PDP Slides
    • Affects all applications that will be used in WindRiver
    • The idea in FP level is that only subsystems that are needed by product lines using Octeon/PPC are migrated. Action Point: Find out which features and protocols are tested in Windriver OS.
    • Applications must be endianness-neutral, which requires test compilations and preferably manual inspection of the code
    • This is change in compilation environment so the available libs etc. may change. Early test compilations suggested.

[edit] Status of old subsystems in Cat

  • Subsystems very unlikely to be needed:
    • SS_TestDp: No Dedicated Port load balancer in Cat
    • SS_TestJMSmon: Has not been used in FP for a while.
    • SS_TestTCP: : No TCP load balancer in Cat, and if there is it will be tested with SS_TestSTG or external generator
    • SS_TestCOPS: No COPS support
  • Subsystems unlikely to be needed:
    • SS_TestBIT: PV has not indicated that they would still like to use HIT & Telnet
    • SS_TestCORBASec: Testing Corba marked as not relevant in Test Environment Matrix
    • SS_TestIPD: This might be useful, but it has not been used by anyone
    • SS_TestLink: Specific solution to UAN-10 deployment in Charm
  • Subsystems likely to be needed:
    • SS_TestALARM, SS_TestAlarmAgent: Needs re-writing. These subsystems should be combined to form new SS_TestAlarm.
    • SS_TestDIA
    • SS_TestHAS: Except for java parts
    • SS_TestNFS
    • SS_TestPM
    • SS_TestRUIM
    • SS_TestSNMP
  • Subsystems already fixed for Cat
    • SS_TestCPUMEM: (Charm functionality. Not able to measure much in Cat)
    • SS_TestSCCPuser
    • SS_TestSIP: Might be needed as a general-use TCP generator, altough more versatile application should be used for this purpose.
    • SS_TestSTG: Requested by PV. Some other application with similar functionality is also sufficient.
    • SS_TestTCAPuser
    • SS_TestUtil

[edit] Concrete actions

[edit] Internal Release 0

  • We must take an active role in getting requirements and information from Integration, PV, and SyVe early enough.
  • The work needs to be ready when testing IR1 starts. It is therefore not mandatory to start implementing these changes before IR0 is available, except maybe for TestFICM, and possible new Octeon applications.
  • No changes should be announced before reviewed and approved.
  • RTE changes
    • The changes affect all applications that will be included in Cat
    • The changes are relatively small and they can be implemented when we have IR0 available for testing.
    • Exact changes are easier to plan and test when we have IR0 available.
    • Default log file path changed to /var/log/SS_Test<name>.
    • New application must use LDAP for storing configuration.
    • Existing applications have some alternatives for handling configuration:
      1. Application is modified to read its config directly from LDAP. CppPrefs could be used.
      2. Configuration is stored in LDAP and an activation plug-in is created, that generates the configuration file each time LDAP is modified. Simple activation plugins are Bash scripts called translation scripts.
      3. Configuration stays in a file. A file containg the config file path must be added to /etc/gstatetab.d directory. Legal place for config file TBD.
    • It must be investigated which changes can be implemented in SS_TestUtil's side.
    • State data files must stored to node-local or global state volume and must be listed in /etc/statetab.d or /etc/gstatetab.d, respectively.
    • Name of the cluster can be read from CLUSTERID env variable.
  • Update SS_TargetInfo for Octeon Rackmount
  • PPC
    • The changes affect all applications that will be included in Cat. More evaluation can be done when test plans are available.
    • Following Flexman's instructions make sure that the application (cross-)compile and that they don't assume anything about platform's byte order.
  • Verify that our servers work in Octeon Simple Executive
    • This is to be done only based on test plans and requirements when those become available
    • It is likely that this is not needed for any appication.
    • Following Flexman's instructions make sure that the application (cross-)compile and then test in the target environment.
  • New subsystem
    • SS_TestFICM
      • Project managament must follow up that this task is done in a sensible manner and duplicate work is avoided.
    • Universal Traffic Generator
      • Possibility to include a good-quality open source application<ref name="seagull">For Example Seagull</ref> in the build to replace in-house traffic generators should be investigated.
