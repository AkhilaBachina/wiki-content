TCExec2 requirements

From SugarWiki
Jump to: navigation, search

This page is a draft for the requirements document of TCExec version 2. See TCExec2 Design Draft for the plans how to fulfill these requirements.

Requirements from user's point of view

  • TCExec2 must be able to run one test case per execution (No need to run multiple test cases like TCEexec1)
  • Test case execution is described by testcase script that can be self-sufficient (No separate configuration file and test case file as in TCExec1).
  • Test case script must be divided in logical blocks that can be nested.
  • User must be able execute auxiliary scripts implemented in any scripting language (bash, perl, python, ruby etc.) from testcase script.
  • User must be able to define variables in the scope of enclosing block in the test case script.
  • User must be able to export variables from auxiliary scripts to block enclosing the command to execute the script
  • TCExec2 must be able to execute tests against any kind of target. Protocol used for connection must be abstracted. The target might be for example some router or switch. In more bizarre cases the target might some GUI or WebUI which we are 'scripting' using some GUI testing plugin.
  • TCExec2 must be able to use arbitrary number of targets in one test case.
  • TCExec2 provides CLI.
  • TCExec2 creates customizable report from test run.
  • TCExec2 must be able to parse TCExec testcase. TCExec configuration parsing not needed.

Requirements from FlexiTAF point of view

  • FlexiTAF uses TCExec2 programmatically with Java without spawning external processes.
  • FlexiTAF receives result object from TCExec2 after execution.
  • FlexiTAF can inject variables to TCExec2.
  • FlexiTAF can inject proxy objects to TCExec2. Proxy objects can be used in testcase logic and scripts to control for example protocol generators.

Design requirements

  • Testcase scripts are implemented in XML, which control the testcase logic. In XML script we can then run auxiliary scripts.
  • Has core library
    • Testcase-builder-api with implementation to read TCExec testcases.
    • Transport-api with default protocol implementations
    • Result-api, with default implementations to create HTML and txt reports.
    • Plugin-api, can be used to declare new keywords for testcase logic XML.
    • Scripting-api with default provides for well-known scripting languages. Should also support that testcase can depend of some library (for example some python module) which would be then automatically transferred to the target where scripting environment needs it.
    • Executor which is responsible of executing testcase
      • Main logic always runs at the server where it was started
      • Executor can distribute itself to targets and start new JVM there. Distributed executors can talk with each other using RMI for example. This is required if we want to support proxy object injections.
  • Has CLI module
  • For FlexiPlatform we would create default set of plugins which can be used to ease up Flexi testing.
