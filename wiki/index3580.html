[edit] Preface

This Wikipage shall provide information about how we (TFRM) think that FT testcases should be designed and implemented using FlexiTAF. The following guidelines are based on the General Principles for test automation and the Implementation Guidelines but shall provide more examples.

[edit] Development of testcases

[edit] How to start a new testcase

A new testcase should normally be started from scratch. Use afcli to create the testcase directory, an empty testcase metadata file, and empty tc and tc_analyze files.
Example: 

afcli –-release 1 –-exec-engine tcexec create testcase ExampleDomain/ExampleSubdomain example_001

This command creates a new testcase example_001 in release Release_1, domain ExampleDomain subdomain ExampleSubdomain.

[edit] How to create a testcase which is similar to another testcase

It should always be avoided to just copy an existing testcase and then start modifying. If a testcase (e.g., example_002) has very similar behavior than another testcase (e.g., example_001), the functions used in example_001 should be analyzed if they can (and will) be used by several testcases in the domain/subdomain. If yes, these functions should be copied to the common directory of the domain/subdomain, respectively. All testcases which then want to use these functions, should add appropriate copy commands in their testcase metadata file ("testcase.xml").
Never use symbolic links from the testcase's tcexec directory to the common function.

[edit] Process of developing/maintaining a testcase

  1. When developing a new testcase, create the (empty) testcase as described here: How to write good testcases#How to start a new testcase.
  2. For existing testcase, do "svn update" to really edit the current version of the testcase. (Note: Do "svn update" always on highest level to make sure that your working directory contains also the latest version of all common functions and domain metadata files.)
  3. Then implement/change the testcase logic and test it on appropriate target using the --local option of afcli.
  4. Once the testcase is finished, add the files to the repository (for new testcases) and commit it. Check with "svn status" that all files which belong to the testcase are in the repository. (Note: Although it is preferable that the testcases is passing before it is committed, the implemented but failing testcase can also be committed if the tested feature is not working -- as proven by the testcase.)
  5. Finally, do (at least) one execution without the --local option to (a) have an official test case execution result for reference and (b) to make sure that really everything has been committed.


[edit] Test case coding guidelines

  • no hard-coded IPs, usernames, passwords in the tc file or other functions
  • define testcase-wide variables in the testcase's metadata file
  • extensively use metadata tasks wherever possible/applicable (for exact syntax see FlexiTAF User Guide)
  • use TCExec exit codes correctly (see TCExec Exit Code Guide)

[edit] Common functions

[edit] How to use common functions

Common functions/scripts should be put to the domain's or subdomain's common directory (or any subdirectory there), respectively. If a testcase wants to use any of these functions/scripts, appropriate copy commands should be added to the testcase's metadata file ("testcase.xml"). If all testcases of a domain/subdomain use the same set of common functions, the copy commands can also be given in the respective domain's metadata file ("domain.xml").

Never use symbolic links form the testcase's directory pointing to file containing the common function/script. Testcases using symbolic links should be changed such that they are replacing the symbolic links by copy commands.
The rationale behind this

  1. Files (and also symbolic links) in the Subversion repository are checked out in random order. If there are executable symlinks, checkout fails if the target of the symlink is not yet checked out when the Subversion tries to set the executable flag to the copy of that file. (This is a feature of Subversion and propably won't be fixed anytime soon. Only solution is just to avoid symlinks.)
  2. Symbolic links in the testcase directory are difficult to distinguish from test-specific files if one debugs all the files in the testcase's tcexec directory (without always carefully checking if it is test-case specific or just a symbolic link). Then, common files are accidentally changed in a way that other testcases fail.

[edit] Examples of copy tasks in the testcase's metadata file

To copy all files with ending .sh which are in the subdomain's common directory (subdirectory script) to the testcase's tcexec directory: 

<testcase model-version="1.0.0" description="">
     <tasks>
         <pre>
             <copy>
                 <fileset path="../common/script">
                     <glob pattern="*.sh"/>
                 </fileset>
             </copy>
         </pre>
    </tasks>
 </testcase>

[edit] Examples of copy tasks in the domain's metadata file

To copy two specific files from the domain's common directory (subdirectory script) to the testcase's directory example_001/tcexec/script/ 

<domain model-version="1.0.0" description="">
     <tasks>
         <pre>
             <copy dest="script">
                 <file path="common/script/CommonFunctions.sh"/>
                 <file path="common/script/OtherFunctions.sh"/>
             </copy>
         </pre>
     </tasks>
 </domain>

[edit] Coding guidelines for common functions

Common functions are supposed to be used by several testcases. Thus, it is mandatory to provide appropriate documentation with each function. At least provide the following:

  • brief description of the common function's functionality
  • list of input parameters and their possible values
  • list of output parameters
  • possible return codes and their meaning
  • other common functions used by this function (the function's dependencies)
  • comments/hints/tips
  • author and date of the implementation

[edit] Which common functions are available

In addition to common functions of specific domains or subdomains, it is also possible to have common functions for all domains.

From time to time, the existing domain/subdomain-specific common functions should be assessed if they are of global use. Then, they should be moved to the general common functions. (Note: This requires to update the copy tasks in the metadata files!)

[edit] Test applications

[edit] Test applications provided with the (testing) build

[edit] How to use specific self-provided test applications

Self-provided test applications should be stored in a subdirectory of the testcase (if the application is only used there) or the domain/subdomain (if the application is used in several testcases). The test application should comprise the header files, the source code as well as makefiles that can be used with the respective build environment (e.g., LinSEE). All testcases which want to use the test application should add a build task in their metadata file ("testcase.xml"). If all/many testcases of a domain use the test application, the build task can also be in the domain metadata file ("domain.xml").

To build a C++ test application called "My test application" on LinSEE with clearmake, the following build task has to be added: 

<testcase model-version="1.0.0" description="">
     <tasks>
         <pre>
             <build name="My test application" provider="linsee"  make-prog="clearmake" make-args="-C gnu" makefile-path="common/TestApp/build/makefile">
                 <input>
                     <fileset path="common/MyTestApp/src">
                         <glob pattern="*.cpp"/>
                         <glob pattern="*.h"/>
                     </fileset>
                     <file path="common/MyTestApp/build/makefile"/>
                 </input>
                 <output>
                     <file path="common/TestApp/src/test_app"/>
                 </output>
             </build>
         </pre>
     </tasks>
 </testcase>

[edit] Other

[edit] How to do LDAP changes

If possible, LDAP changes should be done using FlexiTAF LDAP tasks which are defined in the metadata file (testcase or domain metadata file). These LDAP tasks can be performed before the tc file of the testcase is executed (pre phase) and also after its execution (post phase).
Since a testcase should leave the target cluster in the same condition as it has been before the testcase execution, remember to revert your changes:

  • Adding entries in the pre phase should be accompanied by removing that entry again in the post phase.
  • Changing entries in the pre phase should be accompanied by changing that value back to its old value (if known).
  • Entries removed in the pre phase should be added again in the post phase (if these entries are usually there in LDAP).

[edit] Adding LDAP entries

New LDAP entries can be added using the FlexiTAF LDAP task "ldap add" in the metadata file (testcase or domain metadata file). An example is provided in the FlexiTAF User Guide.

[edit] Modifying LDAP entries

Existing LDAP entries can be modified using FlexiTAF LDAP task "ldap modify" in the metadata file (testcase or domain metadata file).
The following example enables an (imaginary) ping services that is started by setting the respective entrie's attribute "enablePingServer" to "true". The LDAP entry is modified before the testcase is executed (pre phase) and also after its execution (post phase) to set the value back to "false".
Note that this example does not restore the old value which is not good practice (and can cause problems in other testcases). However, this problem can currently not be solved in a better way using ldap tasks. 

<testcase model-version=”1.0.0” description=”Test Local Ping Service”>
     <tasks>
         <pre>
            <ldap>
                <modify dn="PingServer=testPingServer,
                        FragmentId=PingServer, targetId=TargetRoot">
                    <attribute name="enablePingServer" value="true"/>
                </modify>
            </ldap>
         <pre>
         <execute>
            <tcexec timeout=”10”/>
         </execute>
         <post>
            <ldap>
                <modify dn="PingServer=testPingServer,
                        FragmentId=PingServer, targetId=TargetRoot">
                    <attribute name="enablePingServer" value="false"/>
                </modify>
            </ldap>
         </post>
     <tasks>
 </testcase>

[edit] Deleting LDAP entries

Existing LDAP entries and LDAP entry attributes can be removed using the FlexiTAF LDAP task "ldap remove" in the metadata file (testcase or domain metadata file). Removing is non-recursive so only leaf-nodes can be removed.
An example is provided in the FlexiTAF User Guide.


[edit] Importing LDAP entries

Modifications can also be done directly using standard LDIF and the LDIF import task in the metadata file (testcase or domain metadata file). 

<ldap>
   <ldif-import>
     <file path=”my_changes.ldif”/>
   </ldif-import>
 </ldap>
