General

Backup

  • Common backup scripts stored to /twafs/backups/common
  • Server specific scripts invoked from /etc/cron.daily, /etc/cron.weekly
  • Each type has own backup scripts. Currently scripts exists for "mysql", "server", "svn" and "trac". Type specific backup scripts are stored to /twafs/backups/<type>/script.
  • Backups are written in /twafs/backups/. Each type has own directory there /twafs/backups/<type>/backup.
  • Backups for each type use same rotation where
    • Daily backups from current month are stored to /twafs/backups/<type>/backup/current
    • Weekly backups are stored to /twafs/backups/<type>/backup/weekly
    • Monthly backups are stored to /twafs/backups/<type>/backup/monthly
  • Rotation scripts don't delete any data. It just moves all deletable backups to /twafs/backups/archives/<type>/archive.

AAA

  • User credentials are stored to NIS at odyssey.fp.nsn-rdnet.net
  • Current user groups have following logic
    • twaroot (1200) is for admins
    • twausers (300) is for normal FP users
    • games (20) is for users outside FP for limited access

Compilation of new application from sources

  • When you need to compile something from sources because RPMs are not available remember to always set --prefix /opt/<appname>-<version> to "./configure".
  • In some cases you need to make dynamically linkable libs also visible to the system. Add needed dir to /etc/ld.so.conf and rebuild the cache "ldconfig -v".

Sugar

Apache httpd

  • Control with init scripts service httpd ...
  • Config file is /opt/apache-2.2.3/conf/httpd.conf
  • User credentials are read from /etc/odyssey_passwd and /etc/odyssey_groups
    • These files are updated by /etc/cron.hourly/ypcat.cron

MediaWiki

  • Config file is /wwwroot/wiki/LocalSettings.php
  • Database is in mysql called wikidb
  • Wiki user Root has admin privileges
  • To promote yourself to admin you need to add yourself to the sysop-group. You can do this by adding entry to the user_groups table in database.
  • Additional css applied to all pages MediaWiki:Common.css

PostgreSQL

  • Control with init scripts service postgresql ...
  • Data files stored to /usr/local/pgsql
  • Always use PostgreSQL database as user postgres
  • Admin command is psql

MySQL

  • Control with init scripts service mysql ...
  • Data files stored to /var/lib/mysql
  • Use "reset master" command to purge old binary logs. This is required to be done periodically to free some diskspace.

Trac

  • Trac information is stored in '/usr/local/trac'
  • Use trac-admin command as root in sugar

Subversion

  • Data files stored to /usr/local/svn
  • No separate init scripts for testcases and tfrm repositories. Front-end is loaded with httpd.
  • SS_FlexiTAFTC_mirror is mirrored from iSource with /etc/init.d/svnmirrord. If svnsync prosess is killed in the midle of sync, it leaves lock property to the repository which must be manually removed (as root) with svn --username twaroot propdel -r0 --revprop svn:sync-lock file:///usr/local/svn/SS_FlexiTAFTC_mirror/.

FlexiTAF Server

  • Control with init scripts service flexitaf .... /etc/profile defines the version which init scripts will start.
  • Databases in mysql are called flexitaf_jobs, flexitaf_webui, flexitaf_result, flexitaf_target
  • Installed to /opt/flexitaf-server-<version>
  • /wwwroot/secure contains symlinks pointing to the installation directory. These needs to be updated when new flexitaf-server release is installed.
  • Local copy of testcases is placed to /opt/flexitaf-server-<version>/work/repository/testcases. twauser account is used to checkout. For external links to SVSS testcases a4virtan credentials are used.

Build providers

  • Check used build provider host from flexitaf configuration in sugar. ClearCase build provider names start with prefix CC- and SVN providers with SVN-.
  • Sanman-account is used in Linsee for running providers
  • Installed to /home/sanman/flexitaf/flexitaf-server-<version>
  • Before using starter scripts following scripts needs to be sourced: source /home/sanman/flexitaf/env_linsee_cc.sh for ClearCase servers and env_linsee_svn.sh for SVN servers.
  • The folowing steps also should be executed before starting the buid-provider.
    • Modify the PIDDIR variable in flexitaf script (/home/sanman/flexitaf/flexitaf-server-<version>/bin/linux-x86-32/flexitaf)
      • PIDDIR="./${BUILDPROVIDER_INSTANCE_ID}"
    • Create a directory inside /home/sanman/flexitaf/flexitaf-server-<version>/bin/linux-x86-32/.The directory name should be <type>_<host>.
    • Edit /home/sanman/flexitaf/flexitaf-server-<version>/conf/wrapper.conf
      • set.default.JMX_ENABLED=false
      • wrapper.java.additional.7=-Djava.util.logging.config.file="%FLEXITAF_HOME%/conf/%BUILDPROVIDER_INSTANCE_ID%.logging.properties"
    • Create a link from conf dir to flexitaf-server-conf directory
      • cd /home/sanman/flexitaf/flexitaf-server-<version>/
      • rm -rf conf
      • ln -s ../flexitaf-server-conf/ conf/
  • Build provider can be started with ./home/sanman/flexitaf/flexitaf-server-<version>/bin/linux-x86-32/flexitaf start.
  • PID file is written to /home/sanman/flexitaf/flexitaf-server-<version>/bin/linux-x86-32/<type>_<host>/<pidfile>. For stopping build provider you can use: ./home/sanman/flexitaf/flexitaf-server-<version>/bin/linux-x86-32/flexitaf stop.
  • Adding new build provider instance requires that you create correctly named configuration xml file and logging properties file to installation conf directory.
  • NOTES: 
   1)If stopping of the buildprovider fails for some reason, before starting it you need to take care that it isn't yet running ps axu|grep wrapper|grep sanman. If it is just kill the old process and then start it.
   2)A different set of build providers are running on all the required LINSEES for BlrTAF . The installed packages & the env scripts for them are located in the path /home/sanman/flexitaf/BLR/
