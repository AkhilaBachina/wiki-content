[edit] Grosso commissioning interfaces

I had to remove the existing IP addresses for FEWS' commissioning interface eth1 and have added another IP address for commissioning of IBM BC. When FEWS is used again for commissioning PoC these IP addresses have to be set back according to commissioning quick guide (or as seen below).

Easy way to switch back to PoC interface configuration: 

ip a d 192.168.0.250/16 dev eth1    # removes the configuration needed for IBM BC commissioning
ip a a 192.168.0.250/24 dev eth1    # adds configuration for NB commissioning cable
ifconfig eth1:0 10.0.0.250 broadcast 10.0.0.255 netmask 255.255.255.0     # adds configuration for NB maintenance cable (to switch)

Note: The latter interface and IP address is only necessary if you are using maintenance cables to connect to the NokiaBlade switches (as it is, for example, required when doing automated commissioning).

[edit] To do first time start for WASPlatOM RG

  • If the WASPlatOM fails to restart or or gets stuck at Initializing state after a restart then the whole RG needs to be done a first time start up. During which the RG will do all the set up from the scratch. Though this will take around 20 minutes RG will become active.
  • Steps to be done.

1. Execute the below command 

touch /var/opt/Nokia/WASPlatOM_1st_time_start.txt file

2. Restart the RG WASPlatOM 

hascli -rn /WASPlatOM
  • This will start the WASPlatOM RG from start and should normally get started fine.

[edit] Ping to Switches fails during commissioning

  • In an FP4 environment, commissioning was failing at the step where the switches in the cluster were verified using ping.
  • The ping to switches in the right side of the Disks were always failing.
  • On investigation it was found the problem was due to wrong fibre channel and ethernet cabling in the cluster.
  • The correct cabling and other configurations were obtained from Flexiserver reference configurations document(Link provided in the main page).

[edit] Tool to plot memory and CPU usage graphs

The script will be useful for plotting the memory usage and CPU usage specially for long term stability tests. The details about the script is mentioned below. The memory usage and CPU usage values are retrieved from the result files of the TestCPUMEM tool.

Usage:: sh longterm.sh <Cluster IP address> <Writing frequency> [X-axis interval]

  • Cluster IP address::Directory IPAddress
  • Writing frequency:: The value configured in TestCPUMEM config file
  • X-axis interval:: Values at the specified interval will be picked and plotted in the graph.
