[edit] Introduction

Test applications are real production software. They must work in reliable, predictable and well documented manner. Unfortunately, test applications are sometimes initially developed thinking only one testing phase, like module testing, and one user, like the author him/herself. Usually the same applications are needed in other test phases, and even by other products later on. They are also used and updated for maintetenance testing long after original "final" release.

Re-writing and improving applications by different people consumes huge amout of resources. In the long term it saves effort for everybody to create good quality in the first place. But don't be afraid to release the code when it is "good enough" for current needs.

[edit] Basics

The requirement level for the applications depends on its intended use. If there is no reason the decide otherwise the test applications should be developed stability testing in mind.

[edit] Quality requirements

  • Make code readable by others. Also the test applications must be easy to debug and fix if and when problems arise. Write comments. Don't use cryptic variable and function names.
  • Remember the quality. The better the design the easier it is to correct problems and add features later on.
  • Don't be too clever. Write code that can easily be undestood and modified by others. Document complex and novel solutions clearly.
  • Think of the users. The applications must be intuitive to use. They must detect errors and provide help. Segfault is not a proper response for a typo in the configuration file. Application output must be such that user can verify that the feature under test is working properly, or if the feature is broken the test application must provide helpful diagnostic messages.

[edit] Stability testing requirements

  • Applications must survive running for two weeks without interruption. In the meantime the library or the feature that the application is testing can be stopped and started many times or it may crash. Test application cannot get confused by this.
  • The application must provide meaningful result logs with reasonable amout of data during the long execution. It's not allowed to print results only at the end.
  • The test application footprint must be as small as possbile; Leaking memory or waiting for random event in a busy loop is not acceptable. Nobody wants to see a test case to fail after 10 days because the test application causes the system to crash.

[edit] General Rules

[edit] Files, Building, Distribution

  • Everything has to be stored in version control
  • Application should be distributable as a single package
  • All releases have some unique version number which is tagged in used version control. The code of tagged release shall never change.
  • Application should be buildable with single command
    • You should use some make-system like Ant [1], Make, Clearmake [2], Automake, Maven [3] etc. Ant is very easy to use and can be also used for c++ applications.
  • Application should be startable with single command
  • Application should be packaged/released with single command
    • You should provide some simple start script or binary which can be started.
  • Use issuetracker for documenting defects and improvement ideas during development and maintenance [4]

[edit] Design

  • Prefer UML diagrams when writing designs. Visio and Rational Rose are good tools for creating UML-diagrams.

[edit] Implementation

  • Utilize well known libraries
    • C++:
      • Use STL for everything you can
      • If something is missing from STL, check Boost [5]
      • Use TCLAP for command line argument parsing API docs
      • Use LibTFRM for some specific things API docs. It provides mostly helper classes for common problems encountered during test application development (how to handle input/output, reporting, configurability, message sending loop etc.).
      • Use Xerces for XML handling (LibTFRM has helper classes for Xerces) XercesC page
      • Use threading library for threads (pthread, zthread ZThread page etc)
      • Order for selecting implementation to specific problem: STL -> Boost -> Other 3rd party libraries -> LibTFRM -> implement it by yourself.
    • Java:
      • If you think something is missing from Java Standard Library check Jakarta Commons libaries Jakarta Commons page
      • Use Jakarta Commons CLI for command line argument parsing Jakarta Commons CLI Page
      • Use Xerces for XML handling Xerces page
      • Use tfrm-util for some specific things [6]
      • Order for selecting implementation to specific problem: Java standard library -> Jakarta commons -> Other 3rd party libraries -> tfrm-util -> implement it by yourself.
    • Perl:
    • Python:
      • Use standard library (modules) distributed with Python [7]
    • PHP:
      • Use PHP standard library
      • Use PEAR libraries [8]
      • Use Zend Framework [9]
    • Bash:
  • Always avoid hardcoding. No hardcoded paths, addresses, message contents etc. Prefer configurability.
  • Never write directly to stdout, use some logging-system instead. Not required for small applications. Use common sense.
  • Application should be always able to shutdown gracefully (release all resources)
  • Configuration files should be done in well known formats (for example XML or normal INI-file format)
    • LibTFRM also provides parser for one format (similar than INI-file formats) which is used in existing Test Applications.
  • Whenever you notice that you are writing some general purpose code, separate the implementation so it could be easily moved to external library (for example LibTFRM or tfrm-util).

[edit] Command line input

  • If application accepts some command line input, following options should be used:
    • -h, --help: Print help
    • -d [level], --debug [level], --verbosity [level]: Set verbosity
    • -f [path], --file [path]: Configuration file
    • -v: Print version information (see also FP specific comments below)

[edit] Output

  • Application should provide reasonable error, help and usage messages in every case.
  • Application must use exit code to signal succeess or failure.
  • Applications should be able to write logs to user-defined place. Simple applications can achieve this simply by letting user to redirect all output to somewhere.
  • If application provides complex results, these should be written in well-known format (CSV or XML for example). XML in some cases is better alternative if application produces huge amount of data since we can then use XQuery to select smaller subsets for analysis.
    • LibTFRM has own OutputManager implementation which support output formatting using different formatter and printer objects.

[edit] Testing

  • Implement unit tests
    • C++:
      • CppUnit [10]
    • Java:
      • Guidelines how to write unit tests [11]
      • JUnit [12]
      • TestNG [13]
      • JMock [14]
  • Unit testing process should be integrated to used build-system
  • Implement unit tests which can be easily executed without complex environment setups.
  • Larger applications which consist of multiple modules should also have easily executable integration tests.
  • Use static code analysis tools
    • Java:
      • Checkstyle [15]
      • CPD [16]
      • PMD [17]
      • FindBugs [18]
      • JDepend [19]
      • JavaNCSS [20]
  • Use dynamic analysis tools
    • Java:
      • Cobertura [21]
  • For environment where you don't have garbage collector available you should ensure that there are no memory leaks using proper tools

[edit] Documentation

  • If you are developing library, document sourcecodes (class and functions) so API docs can be generated
    • C++:
      • API docs can be generated with Doxygen [22]
    • Java:
      • API docs can be generated with Javadoc [23]
  • Write a design document. (template)
  • For documentation use NSN_DocTemp_DINA4_basic_numb_col_duplex template. You can find it from brand media downloads in Sharenet-IMS: Word templates

[edit] Version control usage

  • Always commit things mapped to some existing issue. Use commit messages for documenting the issue commit is addressing.
  • The commit message should always begin with issue number. For example: 
#245: Fixed bug.
  • In some cases we might just add some comment to sources which doesn't really affect program behavior. In those cases it is valid to commit things without adding any issue number. Use common sense: 
Fixed typo from comments.

Implemented new unit tests.

Fixed maven plugin configuration.
  • It is valid to do multiple commits for single issue: 
#443: Simplified database queries.

#443: Added new utility class.
  • If you are doing refactoring which doesn't affect program behavior (like refactoring shouldn't) and the task is rather small, it is allowed to commit changes without mapping it to any open issue: 
Added new constructor.

Renamed method and some varibles.

Optimized result calculation.
  • If you are doing refactoring at the same time when you are implementing some open issue, then associate commits with that issue number.
  • Avoid addressing multiple issues with single commit: 
#443, #322, #245: Fixed tons of bugs.
  • If you are merging things between branches use following comments: 
Merged r<REV-ID>: Fix for bug #456

[edit] FlexiPlatform specific applications

[edit] Software Engineering Environment

  • Application development is done in LinSEE (LinSEE Servers, Official LinSEE Page) using FlexiPlatform specific activity scripts. (SCM Instructions)
  • Applications are stored to Clearcase or SVN under proper FlexiPlatform branch with name SS_Test<AppName>. These branches are created by flexman (Flexman SCM Pages).

[edit] Document templates

  • TA Requirement Document Template
  • TA User Guide template
  • Design Document Template (same link as in the previous Chapter of the Guide)

[edit] Generic rules

  • HAS-aware applications should be usable also without HAS
  • Consider using Flexi specific libraries for some functionalities like CppLog for logging
  • Follow the RTE specification for Charm, for Cat

[edit] Deployment rules

  • Create deployments for the applications which are service-oriented and can operate alone at the background. Creating deployment for some CLI tool isn't useful.
  • By default single RG should be enough for each application.
  • Create multiple RUs for applications where they are useful. For example if application can operate either in client or server mode, then create one RU for each. Also if the application supports multiple protocols (like Diameter) then it can be useful to also separate different default protocol specific configurations under separate RUs.
  • Create separate entry to processtypes.ldif for each RU. Each instance should use configuration file from different location. For example "FSTestDiaReceiverTCPServer" and "FSTestDiaSenderTCPServer" would use configuration files from paths "/etc/opt/Nokia/SS_TestDia/TestDiaTCPReceiverServer.xml" and "/etc/opt/Nokia/SS_TestDia/TestDiaTCPSenderServer.xml". Configuration file name can be quite directly mapped to RU-name, to make it easy for the user to understand which config-file affects to which RU.

[edit] Naming convention

  • RG: <TestAppName>. For example: 
/TestSNMP
  • RU: FS<TestAppName><Mode>. For example: 
/CLA-0/FSTestSNMPServer
/AS-0/FSTestDiaTCPReceiverServer
/AS-0/FSTestDiaTCPSenderServer
  • Process: <TestAppName>. For example: 
/CLA-0/FSTestSNMPServer/TestSNMP

[edit] Usage examples

  • Start TestFICM server at AS-0 and client at AS-1: 
hascli -u /TestFICM /AS-0/FSTestFICMReceiverServer /AS-1/FSTestFICMSenderServer

[edit] Input/Output

  • Applications started with HAS either must not print to stdout or stderr during normal operation, or they must provide a command line option -q, --quiet that supresses all output.
    • Other applications should follow this practice.
  • Applications should provide command line option -d<level>,--debug=<level> for printing log messages to standard output.
  • Preferred location for the application runtime configuration files is /etc/opt/Nokia/SS_TestName/etc/
  • Preferred location for the application log files is (node-local) /var/log/SS_TestName/
    • If there are many instances of the application running in different nodes, and they write separate log files, each log file should have the node name included in the file name: 
 /var/log/SS_TestSIP/TestSipServer-result-AS-0.log
 /var/log/SS_TestSIP/TestSipServer-result-AS-1.log
  • Log files must be opened for writing in append mode.
  • Read-only NFS is used in Charm and in the later builds for system image
    • /etc/opt/Nokia/SS_TestName/ resides in system image
    • Diskless nodes are unaffected. They can write data and do locking as they have used to.
    • Applications running in CLA nodes must care if they write to system image. All writes to system image have to go through _active_ CLAs local sysimg mount point.
    • It is not possible to write to sysimg in inactive CLA (over NFS)
    • Meaning: 
 /var/mnt/remote/sysimg <-- Will be READ ONLY on CLA nodes
 /var/mnt/local/sysimg  <-- Is writable on ACTIVE CLA node

[edit] Version information of test applications

Test applications shall have a command-line parameter -v for printing the version information as it is known in the used version control system.

  • SVN: Use for example "%Id%" (more info in SVN manual)
  • Clearcase:
    • For textfiles, follow the instructions given in the RPM grannyguide for FlexiServer product family
    • For executables, this approach is not working. Simplest possible workaround is having a version.txt file that gets updated and is then referred or read+printed. Another workaround could also use the information which is already in the executable file and can be seen using "objdump --section=.versionlabelinfo -s <my_executable>".

[edit] Non-Flexiplatform specific applications

[edit] Software Engineering Environment

  • Applications are stored to our own Subversion server (svn root: http://sugar.fp.nsn-rdnet.net/svn/tfrm)
    • New directory should be created in the root of the repository for larger applications.
    • Small and one-shot scripts should be stored under Scripts directory.
  • flexiaf5 (that is the FlexiTAF development host) can be used for generic application development.
    • Don't use flexiaf5 for CPU-intensive testing, or for any purposes that use resources like sockets or IO extensively.

[edit] Traffic generators

Traffic generator is catch-all term for all test applications that run for a while, generate or listen to some kind of messages and measure the traffic. Examples: Plain TCP client/server (SS_TestSIP), Protocol specific tool (SS_TestDIA), application for generating log messages/alarms/notifications (SS_TestSNMP).

All traffic generators must be stable and reliable enough for stability testing. It must be taken into account that the application may run for hundreds of hours and during that time it may generate billions of messages. The traffic generator is not the only application running in the target during the test.

[edit] Input

  • Application must be configurable.
    • Send rate must be configurable. Measuring only maximum throughput in a busy loop is not sufficient.

[edit] Output

  • Application should write separate status log and result log.
  • Status log is echoed to standard output if a command line option -d <level>, --debug=<level> is given.
  • Result log is echoed to standard output if a command line option -r <level>, --result=<level> is given.
  • Each line in a log file must start with a timestamp.
    • Format: yyyy-mm-dd hh:mm:ss (Simplified ISO 8601, date +"%F %T").
    • Fractions of second can be appended if finer granularity is required. They are separated from full seconds by a period mark (hh:mm:ss.sss).
  • When the application terminates, summary of the generated traffic must be printed to the status log.
    • This summary includes information like total number of messages/transactions, total amount of transferred data in megabytes, average message rate, number of reconnections, maximum number of associations, errors per second etc.
  • Result log consists of interesting statistical data that describes the traffic. Lines are printed with configurable fixed interval.
    • Data is organized in fixed-width columns.
    • A header line must be printed in the beginning of data and can be repeated on regular interval.
      • Header printing interval should be configurable.
    • The information on each line should be the measured traffic after the previous line. Output columns with cumulative data should be used with discretion.
    • Printing is allowed only with pre-defined intervals and timestamps (see table below).
  • Allowed printing intervals and timestamps for result log
Output Interval When to Print
1 s hh:mm:00, hh:mm:01 … hh:mm:59
5 s hh:mm:00, hh:mm:05 … hh:mm:55
10 s hh:mm:00, hh:mm:10 … hh:mm:50
30 s hh:mm:00, hh:mm:30
1 min hh:00:00, hh:01:00 … hh:59:00
5 min hh:00:00, hh:05:00 … hh:55:00
10 min hh:00:00, hh:10:00 … hh:50:00
30 min hh:00:00, hh:30:00
1 h 00:00:00, 01:00:00 … 23:00:00
  • Example of status log: 
 2000-01-01 08:00:00 TestAppl starting 
 2000-01-01 08:00:00 Listening to port 1234
 2000-01-01 08:00:03 Accepted connection
 2000-01-01 08:00:03 Accepted connection  
 2000-01-01 08:00:40 Caught SIGTERM. Terminating...
 2000-01-01 08:00:40 Closed connection
 2000-01-01 08:00:40 Closed connection
 2000-01-01 08:00:40 Total numer of messages received: 4000 msgs
 2000-01-01 08:00:40 Total numer of payload received:  145 MB
 2000-01-01 08:00:40 Messages/sec:                     112.22 msgs/s
 2000-01-01 08:00:40 Payload/sec:                      23.234 MB/s
  • Example of result log: 
 time                     msgs/sec    errors/sec 
 2000-01-01 08:00:00           0.6           0.0 
 2000-01-01 08:00:10           0.4           0.0

  • LibTFRM contains classes for producing output that conforms with this definition.

[edit] Reliability

  • Choose datatypes wisely. Ints do overflow. Double cannot be incremented by 1 much longer than int. Long long is usually sufficient for counters.
  • Try to keep the application footprint small
    • Do not use busy loops
    • Test memory leaks
    • Avoid excessive file I/O
    • Do not allocate more memory than necessary
