[edit] Common instructions

[edit] Versioning

  • All TFRM projects must use tfrm-parent as a parent. Otherwise common settings won't get inherited. In some special cases when you cannot put dependency to tfrm-parent (for examples projects which tfrm-parent depends of) then use tfrm-parent-parent.
  • Use 3 number versioning (1.0.0) scheme for components which have sources
  • Use 1 number versioning (1) scheme for components which don't have sources or doesn't support concept of minor/bugfix release.

[edit] Testing

[edit] Unit testing

  • Use unit tests to test the functionality of the module. Unit tests are suitable for testing which requires no setup or setup that can be done without manual work and is portable between environments. Profiles can be also utilized.
  • Unit tests can have dependencies between projects. If some component's logic essentially depends of some other component then be it. Setup will be just a bit more complex. This shouldn't be a problem. In complex cases you can also use mocking.
  • Prefer JUnit4 and use annotations
  • TestNG can be also used: [1], [2]. It is useful tool if you want to run unit tests in parallel with multiple threads.
  • Keep things simple! Having tests more complex than the actual code which it is testing is not the way to go.
  • Unit test should not cause any side effects and they should be executable in any order.
  • Unit tests are not suitable for complex performance/stress testing.
  • Utilize helper classes from tfrm-util (com.nokia.flexi.tfrm.util.test).

[edit] Integration testing

  • Use integration tests only if absolutely necessary. Those usually end up having duplicate code with unit tests and just increases maintenance burden.
  • Currently Maven also has limited support for integration tests and this should be improved with 2.1 release.
  • Check how Geronimo does integration testing [3]. It tests how existing external projects can be integrated with it.

[edit] Plugins

[edit] Clirr

  • Integrated to build lifecycle with tfrm-parent. Can be executed manually with clirr:check.
  • Set version range to property "previous.compatible.version". It is by default "(,${project.version})". For example 
<project ...>
  ...
  <properties>
    <previous.compatible.version>1.0.0</previous.compatible.version>
  </properties> 
</project>
  • For more information about version ranges check version range specs
  • Clirr plugin will print out errors/warnings if it notices incompatibilities. This usually means that you should increment atleast minor version number, when you make release next time. It won't fail build process currently.

[edit] Dependency plugin

  • See dependencies of some component 
mvn dependency:resolve
  • Analyze dependencies 
mvn dependency:analyze
mvn dependency:analyze-dep-mgt
  • See constructed classpath 
mvn dependency:build-classpath

[edit] Changelog plugin

  • Check changelog 
mvn changelog:changelog
mvn changelog:dev-activity
mvn changelog:file-activity

[edit] Help plugin

  • See effective settings 
mvn help:effective-settings
  • See current profiles 
mvn help:active-profiles
  • See effective POM 
mvn help:effective-pom

[edit] Versions plugin

  • Check are there updates available for used depencencies 
mvn versions:display-dependency-updates -Dorg.apache.maven.global-settings=$HOME/.m2/settings.xml
  • Check are there updates available for used plugin depencencies 
mvn versions:display-plugin-updates -Dorg.apache.maven.global-settings=$HOME/.m2/settings.xml

Note that we need to disable the use of internal repository settings. Easiest way to do that is to use own user-level config as a global config.

You may need to delete your own local repository to get to correct results. Maven caches repository metadata files, and the ones which are cached from flexiaf6 won't show the latest info.

[edit] Javadoc plugin

  • To generate aggreate api docs you need to go to the multi-module project and: 
mvn javadoc:aggregate
mvn site-deploy -N

Note: generated docs aren't linked to the project reports so you need to navigate there manually. These instruction will be deprecated when maven-javadoc-plugin issues related to aggregate reporting are fixed.

[edit] Create component releases

  • Note: always use release-plugin version 2.0-beta-8

[edit] Release single component

  • Update POM and remove all SNAPSHOT dependencies.
  • Use Clirr to check API/binary incompatibilities. Based on the results choose how to increment your component version.
  • Prepare release 
mvn release:prepare

In most cases you can answer by pressing <RET> to all questions. 

What is the release version for "TFRM Checkstyle Configuration"? (com.nokia.flexi.tfrm:tfrm-checkstyle) 1: :
What is SCM release tag or label for "TFRM Checkstyle Configuration"? (com.nokia.flexi.tfrm:tfrm-checkstyle) tfrm-checkstyle-1: :
What is the new development version for "TFRM Checkstyle Configuration"? (com.nokia.flexi.tfrm:tfrm-checkstyle) 2-SNAPSHOT: :

You can use "-DdryRun=true" to test it without committing anything.

  • Publish release to internal repository 
mvn release:perform

[edit] Releasing parent artifact individually

  • If you are releasing parent project individually but you don't have separate directory for the POM (for example com.nokia.flexi.tfrm.util:tfrm-util).
  • Otherwise like single component releasing, but (did not work with maven-release-plugin-2.0, use old instruction below): 
mvn -N -Darguments=-N release:prepare
mvn -N -Darguments=-N release:perform

Old manual way if above does not work:

  • Prepare pom.xml for release by removing SNAPSHOT from it's version and fixing SCM links to point to the correct tag directory.
  • Commit (svn commit -m "Prepare the release of <artifact-id>)"
  • Create directory for tag using naming convention: <artifact-id>-<version> to /tags
  • Copy pom.xml there (svn copy pom.xml ../tags/<artifact-id>-<version>)
  • Commit (svn commit ../tags/<artifact-id>-<version> -m "Copy to tag <artifact-id>-<version>".
  • Increment version from pom.xml in trunk and fix SCM links point to trunk
  • Commit (svn commit -m "Prepare for next development iteration)
  • Deploy (cd ../tags/<artifact-id>-<version>; mvn install deploy -N)

[edit] Releasing multi-module project

  • If you are lucky, release multi-module project is as easy as single component. Just do the release at top-level which you want to be the scope of single tag. If you want to have separate tag for parent and each subproject then see previous instruction about releasing artifacts individually.
  • Sometimes multimodule builds release:prepare will fail. When it fails you need to install "real" versions to local repository manually using partially transformed POMs and then continuing release process: 
mvn release:prepare
mvn install --no-plugin-updates -Prelease-debug,default-repositories,flexitaf-dev,flexiaf5-env,mysql-database,ssh-target-rivendell,telnet-target-rivendell,default-updates-policy
mvn release:prepare
mvn release:perform

[edit] Tips & tricks

[edit] Profiles

The common profile configurations are stored to /twafs/utilities/maven/conf which is symlinked to all maven installations.

Following things are defined there:

  • Internal repository configurations
  • External repositories are redirected to flexiaf4
  • Server where deployed artifacts are transfered
  • Profiles for projects under development
  • Profiles for development environments
  • Profiles for 3rd party components
  • Profiles for external hosts used during testing

Additional profiles are inherited from parent projects. These include

  • Profiles to control AOP load-time weaving. Note: this is currently defined in com.nokia.flexi.flexitaf:flexitaf-core:pom. It will be moved to higher level at some point.

[edit] Project specific profiles

Available: flexitaf-dev

[edit] Host specific profiles

Available: flexiaf4-env, flexiaf5-env

[edit] 3rd party product specific profiles

These profiles should be used to activate optional 3rd party dependencies, like specific databases, during testing.

Available: mysql-database

[edit] External hosts

These contain hardcoded protocol specific ips/usernames/password/etc which are sometimes required for testing, like FlexiPlatform clusters for flexitaf testing.

Available: rivendell-target

[edit] Currently reserved properties

Properties:

  • tfrm.database.mysql.root-user.name: Root username for local mysql installation
  • tfrm.database.mysql.root-user.password: Root password for local mysql installation
  • tfrm.datasource.name: Name of the used datasource
  • tfrm.datasource.type: Type of the used datasource
  • tfrm.datasource.host: Host where the datasource is
  • tfrm.datasource.driver: Driver used to connect to the datasource
  • tfrm.datasource.user: Username used for connections to the datasource
  • tfrm.datasource.password: Password used for connections to the datasource
  • tfrm.datasource.url: Url to the datasource
  • external.lib.path: Path where external 3rd party libraries are stored which aren't maven artifacts (like p4a)
  • tfrm.target.ssh.ip: Ip for some target which has ssh protocol available in default port
  • tfrm.target.ssh.user: Username used for connections to the target
  • tfrm.target.ssh.password: Password used for connections to the target

These properties shouldn't be ever used directly in source codes. These are meant to be used in configuration files. If direct access is required check com.nokia.flexi.tfrm.util.test.TestEnvironment and com.nokia.flexi.tfrm.util.test.TestEnvironmentFactory classes which provide accessor methods for those.

[edit] AOP profiles

AOP profiles can be used to weave aspects to your already compiled libraries. Commons aspects are defined in tfrm-aspects projects.

Load-time weaving can be activated to test-phase using test-loadtime-weaving profile. It requires you to provide value for following property:

  • tfrm.test.aop.config: path where aop.xml can be located.

To enable load-time weaving in distribution packages you need to activate it using loadtime-weaving profile. It requires you to provide valui for following property:

  • tfrm.aop.config: path where aop.xml can be located.

For more information about aspects see: [4]

For more information about aop.xml configuration see: [5]

[edit] Trace logging

Trace logging aspect uses normal java logging to print entry and exit logs for method calls.

To enable trace logging you need to provide concrete implementation of AbstraceTrace aspect. It requires you to provide pointcut called traceClass which is used to defined the methods we want to trace. For example if we want to trace all methods from package com.nokia.flexi.flexitaf.repository.testcase we would create following aop.xml: 

<aspectj>
   <aspects>
       <concrete-aspect name="com.nokia.flexi.tfrm.a4virtan.TraceAspect" extends="com.nokia.flexi.tfrm.aspects.logging.AbstractTrace">
           <pointcut name="traceClass" expression="within(com.nokia.flexi.flexitaf.repository.testcase.*)"/>
       </concrete-aspect>
   </aspects>
   <weaver>
       <include within="com.nokia.flexi.flexitaf.repository.testcase.*"/>
   </weaver>
</aspectj>

[edit] Profiling aspect

Profiling aspect uses JETM to measure time spent in specific method calls.

To enable profiling you need to provide concrete implementation of AbstractJetmProfiler aspect. It requires you to provide following pointcuts:

  • profileClass: defines the methods we want to measure
  • profileScopePointcut: defines the scope (entry- and exit-point) of profiling. You have to ensure that program flow won't cause nested calls.

You can control Jetm renderer using following properties:

  • com.nokia.flexi.tfrm.aspects.profile.JetmProfiler.renderer: Selects which output renderer will be used for results. You can use either "text", which is default, or "html".
  • com.nokia.flexi.tfrm.aspects.profile.JetmProfiler.output: Selects path where output will be written. By default it will be written to stdout.
[edit] Unit tests

For example if we want to make individual measurement for each unit test and measure performance of class com.nokia.flexi.flexitaf.repository.testcase.TestcaseCache we would create following aop.xml: 

<aspectj>
   <aspects>
       <concrete-aspect name="com.nokia.flexi.tfrm.ProfileAspect" extends="com.nokia.flexi.tfrm.aspects.profile.AbstractJetmProfiler">
           <pointcut name="profileClass" expression="within(com.nokia.flexi.flexitaf.repository.testcase.TestcaseCache)"/>
           <pointcut name="profilerScopePointcut" expression="execution(@org.junit.Test * *(..))"/>
       </concrete-aspect>
   </aspects>
   <weaver>
       <include within="com.nokia.flexi.flexitaf.repository.testcase.*"/>
   </weaver>
</aspectj>
[edit] Runtime

If we want to measure performance in production environment of class com.nokia.flexi.flexitaf.health.impl.DefaultHealthManager at flexitaf-server side and class com.nokia.flexi.flexitaf.cli.CommandProcessor at flexitaf-cli side we would create following aop.xml: 

<aspectj>
    <aspects>
        <concrete-aspect name="com.nokia.flexi.flexitaf.aspects.ProfileAspect" extends="com.nokia.flexi.tfrm.aspects.profile.AbstractJetmProfiler">
            <pointcut name="profileClass" expression="within(com.nokia.flexi.flexitaf.cli.CommandProcessor) || within(com.nokia.flexi.flexitaf.health.impl.DefaultHealthManager)"/>
            <pointcut name="profilerScopePointcut" expression="execution(* com.nokia.flexi.flexitaf.booter.Booter.main(..))"/>
        </concrete-aspect>
    </aspects>
    <weaver options="-verbose"/>
</aspectj>

Note that we set the scope of profiling to Booter entry-method.

[edit] Tips & tricks

To make life easier you should create your own profiles to your Maven settings.xml: 

 <profiles>
   <profile>
     <id>enable-profiling</id>
     <properties>
       <tfrm.test.aop.config>/home/a4virtan/flexiaf-dev/aop-profiling.xml</tfrm.test.aop.config>
     </properties>
   </profile>
   <profile>
     <id>enable-runtime-profiling</id>
     <properties>
       <tfrm.aop.config>/home/a4virtan/flexiaf-dev/aop-runtime-profiling.xml</tfrm.aop.config>
     </properties>
   </profile>
   <profile>
     <id>enable-tracing</id>
     <properties>
       <tfrm.test.aop.config>/home/a4virtan/flexiaf-dev/aop-tracing.xml</tfrm.test.aop.config>
     </properties>
   </profile>
   <profile> 
     <id>debug-logging</id>
     <properties>
       <java.util.logging.config.file>/home/a4virtan/flexiaf-dev/logging.properties</java.util.logging.config.file>
     </properties>
   </profile>
 </profiles>

And when you want to enable some aspect for your test executions you would just write: 

mvn test -Ptest-loadtime-weaving,enable-profiling
mvn test -Ptest-loadtime-weaving,enable-tracing,debug-logging

Note: you cannot enable both aspects at the same time using this method.

You can also pass extra args for JVM when starting it using property tfrm.test.aop.args. TODO: doesn't work currently.

[edit] Animal sniffer (JDK dependency verification)

Animal-sniffer (https://animal-sniffer.dev.java.net/signature-checker.html) is a tool to verify that no dependencies from wrong JDK ends up into the project. For example we use JDK6 for compilation but we have documented that the projects are JRE5 compatible, which means that compilation will happily pass even we use JDK6 specific stuff, but the end-user is unable to use compiled JARs.

To enable animal-sniffer you only need to enable "sniff"-profile. 

mvn install -Psniff

[edit] Continuous Integration

[edit] Hudson

Hudson is running in flexiaf6 [6]. It is Installed to /opt/hudson-1.214 (TODO: directory name should be changed since version is not anymore same).

Artifacts will be deployed to tfrm.snapshot.repository (flexiaf6:/wwwroot/maven2-snapshot) Project reports are deployed to: /wwwroot/secure/maven-reports/.

Hudson is running as "maven" user. It uses "twauser" account for accessing version control.

Root-user account has same username/password as flexiaf6 root login has.

Data-directory can be found from <INSTALL-DIR>/data.

Hudson has init-scripts for startup/shutdown:

  • service hudson start
  • service hudson stop

To upgrade Hudson you need to:

  • Shutdown Hudson
  • Copy new WAR to <INSTALL-DIR>/lib. The WAR needs to be named "hudson.war" and you overwrite the old WAR.
  • Restart Hudsson

Note that the Hudson uses Maven embedder and it seems that it won't use settings.xml from /opt/apache-maven-<VERSION>/conf/settings.xml. Which means that if global settings.xml is changed at /twafs/utilities/maven/conf/settings.xml you need to copy-paste same settings to /localhome/maven/.m2/settings.xml.

[edit] Continuum

NOTE: Continuum is not used anymore. Hudson has replaced it.

Continuum is running in flexiaf6 [7] using two different schedules:

  • Changed components are compiled and deployed hourly
  • Sites are deployed nightly

Artifacts will be deployed to tfrm.snapshot.repository (flexiaf6:/wwwroot/maven2-snapshot) Project reports are deployed to: /wwwroot/secure/maven-reports/.

Continuum installation has work-directory and build-directory 

/localhome/maven/continuum

When Continuum is updated, new installation should be configured to use that location.

Continuum is running as "maven" user. It uses "twauser" account for accessing version control.

[edit] Repositories

Currently there are following internal repositories:

  • Central mirror (central, flexiaf6:/wwwroot/maven2). Don't deploy anything here by yourself which isn't available at ibiblio.
  • Release repository (tfrm.internal.repository, flexiaf6:/wwwroot/maven2-releases)
  • Snapshot repository (tfrm.snapshot.repository, flexiaf6:/wwwroot/maven2-snapshots)
  • 3rd party (flexiaf6:/wwwroot/maven2-3rd-party). If you want to put some 3rd-party artifact to the repository which isn't available at ibiblio, then put it here.

[edit] Maintenance

Never update dependencies directly to own local repository. Instead download them to internal repository so all developers have same environment, including CI system.

Central mirror is stored to flexiaf6. Change to root user and "su -l maven". Repository is in /wwwroot/maven2/.

[edit] Download new artifact

To download new artifact to shared repository use maven@flexiaf6.fp.nsn-rdnet.net account and 

download-artifact org.codehaus.plexus:plexus:jar:1.0.9
download-artifact org.codehaus.plexus:plexus:jar:1.0.9 http://snapshots.codehaus/maven2

Note that also all dependencies of the artifact will be downloaded.
