[edit] FlexiTAF CLI setup

To get FlexiTAF CLI (afcli) working, you need to source the environment setup script: 

&> source /home/sanman/flexitaf/cli-env.sh (To use sugar as the server)
                    OR
&> source /home/sanman/flexitaf/cli-blr-env.sh (To use BlrTAF as the server)

After that you can use afcli directly from the shell: 

&> afcli list releases

To get CLI configured for your environment, follow the instructions: FlexiTAF_Tutorial#Initial_setup

[edit] SVN setup

In Linsee environment there isn't correct SVN version in the PATH, unless you either run SCM environment script 

&> /build/fvntools/seed/setums.pl

Or the source the environment setup script: 

&> /home/sanman/flexitaf/svn-env.sh

To access testcases using SVN, follow the instruction: FlexiTAF_Tutorial#Testcases

[edit] Testcases in subsystem repositories

Generally it is recommended to put testcases in separate FlexiTAF testcase repository, but in some cases it might be useful to have testcases as part of the subsystem. FlexiTAF specifies certains structure for the testcase repository, so the testcases in the subsystem repository structure for testcases needs to also follow these rules

  • Each testcase has unique name which is specified by the testcase directory name (for example "FT_HAS_SMR_001"). In general the name should be same as the name is for the testcase in TestMan or QualityCenter, to make it easy to locate the testcase design from the management tool.
  • All testcase directories needs to be placed into single directory (domain-directory). For example: 
/SS_SMR
  /tst
    /ft
      domain.xml
      /common   #directory for common files used by multiple testcases
      /FT_HAS_SMR_001
        testcase.xml
        /<tool specific dir: tcexec/fstestcli/robot/selenium/hit/t3>
      /FT_HAS_SMR_002
      ...
  • "common"-directory is the only non-testcase directory allowed to be placed in the same directories with testcases.

[edit] Link subsystem testcases to FlexiTAF testcase repository

With SVN it is possible to link multiple repositories. To get link to your testcases subsystem repository to be added to FlexiTAF repository, you need to contact FlexiTAF keyuser.

Since subsystem tagging/versioning is done differently than in the FlexiTAF testcase repository, it isn't good idea to use the subsystem trunk for any other development than for the latest release. For the older releases all repository links to subsystem trunks needs to be replaced with a link to corresponding subsystem tag. And whenever new version of subsystem is published to the maintenance branch, the link has to be also updated.

[edit] Execute testcases in Linsee

When you have your subsystem testcases linked to the FlexiTAF testcase repository, it is trivial to execute them FlexiTAF_Tutorial#Manage_testcase_executions

You can also checkout the testcases, do local modification for them and debug them locally VersionControl_FAQ.

[edit] Execute subsystem testcases in Linsee during development

During development you haven't yet committed changes, but want to execute the testcases which you have in your svn task. In that case easiest way to do it is to first checkout the testcases from FlexiTAF testcase repository VersionControl_FAQ#How_to_checkout_testcases. You will get following structure (in this case we assume that SS_SMR-domain is the link to the SS_SMR-subsystem): 

/home
  /user
    /svn
      /testcases
        /rel_5.cat
          /FunctionalTesting
            domain.xml
            /HighAvailabilityServices
              domain.xml
              /SS_SMR   #svn:externals link to subsystem repository
                /FT_HAS_SMR_001
                /FT_HAS_SMR_002
                ...
              ...
            ...

If we would now start the testcase with afcli FlexiTAF_Tutorial#Manage_testcase_executions, it would use the version of it which was last time committed. To get local repository to use the testcases from development time svn task, we can delete SS_SMR and put symlink there to point to the subsystem directory: 

rm -rf /home/user/svn/testcases/rel_5.cat/FunctionalTesting/HighAvailabilityServices/SS_SMR
ln -s /home/user/user_20080822190523_cat_has/SS_SMR/tst/ft /home/user/svn/testcases/rel_5.cat/FunctionalTesting/HighAvailabilityServices/SS_SMR

Now when we start the testcase, it will use the implementation from the svn task directory.

Note: be careful not to commit any changes.
