[edit] Build Integration

[edit] BI Introduction

Build Integration is process of verifying that given FlexiPlatform build can be commissioned successfully and certain test criterion for the build are met. It also includes reporting found faults to persons responsible and keeping organization informed about the current BI status. BI usually starts when new implementation is close to finish and ends before actual (functional) testing starts. In iterative model BI has exit at the end of each increment.

In some earlier FlexiPlatform releases BI had been done ad hoc due to missing and vague requirements. Purpose then was to provide build that was mature enough for starting functional testing. Also previously BI was fully manual. This process has lately been specified more thoroughly and part of the BI has been automated (i. e. regression). Build Integration now follows FlexiPlatform Quality Plan and has following exit criterion which has to be met before BI exit:

  1. Done in target HW platforms including virtualized environment and deployments as defined in BI plan
  2. Commissioning completed successfully
  3. Node(s) are up and running
  4. All services are up and running
  5. At least one basic functionality test case executed for all available and critical services
  6. After node restart the node and all services recovers normally to correct state
  7. After all nodes and services are up and running, system must be running 24h without major problems
  8. All open faults reported to Pronto
  9. AN SSH connection can be successfully made every node in the duster
  10. No core files detected in the file system
  11. No log messages classified “CRITICAL” or “Major” are allowed after the first cluster start in the centralized master syslog and in the syslogs on localimgs
