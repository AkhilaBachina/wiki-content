General AutomationFrame related

  • Absolute symlinks should be never used. Instead use relative ones. (e.g. ../../bin/TestDia.cpp)
  • Symlinks between releases should not be done. Instead copy the needed files to the current release.
  • Do not create symlinks to other testcases i.e. FT_TC_001 --> FT_TC_002. This creates duplicate testcases and creates confusion.
  • Testcase directory name has to be always mapped to the testman testcase id.
  • Testcases shouldn't write anything to testcase directory.
  • No hardcoded ip/username/password stuff to the testcases.
  • Domain directories should only contain other domain directories, testcase directories or common directory. Don't put anything else there.

Use Metadata

Use XML-files i.e. subsystem_meta.xml for defining dynamic parameters and performing i.e. test environment set-up. These task can include i.e

  • LDAP changes
  • File copying
  • Building test applications
  • running some command over SSH in other hosts

Look Automation Framework User Guide for exact syntax for these tasks. AutomationFramework User Guide

Write portable testcases

When you define dynamic variables in metadata, you are writing more portable testcases. The FlexiAF will update the values for the variables at runtime and therefore, these testcases can be run in other environment without any changes. If you would have i.e. hardcoded ip-addresses and passwords this would not be possible.

Accessing environment variables in other than bash-scripts

Environment variables can be accessed in scripting languages e.g. Perl, Python through the tools provided by the language in question. This can be done in Perl through ENV hash 

my $home_directory = $ENV{'HOME'};

and in Python using os module 

import os
home_directory = os.environ.get('HOME')

Do not remove testcase directories

Contact any TFRM team member if you have a need to remove testcases directories from SVN. Reason: You can not remove them with rm –r, because also the database entries need to be removed.
