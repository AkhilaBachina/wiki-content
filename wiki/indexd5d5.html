Development Proposals for Automation Frame

[150506-1 a4virtan] Reimplement scheduler to utilize Quartz 3rd party component

Rationale: Quartz is a full-featured, open source job scheduling system. Offers DB backend, transactions and clustering.

Decision: Feasibility study/prototype/update Scheduler design. Effort estimate: Moderate

[150506-2 a4virtan] Implement automatically executable module tests with JUnit for components

Rationale: All components should be tested with automated module tests before new release is published.

Decision: Implementation ongoing.

[150506-3 a4virtan] Implement new component which can be used to do syntatic and semantic changes for existing metadata files

Rationale: If we publish new release which introduces metadata syntax changes which are not backwards compatible we should have a component which can go through all existing metadata files and fix them automatically.

Decision: Design+implement. Effort Estimate: Moderate

[150506-4 a4virtan] Implement new component which updates database

Rationale: If we publish new release which requires different structure of some table in DB we should have a component which updates database.

Decision: Implement. Effort Estimate: Small.


[150506-5 a4virtan] Implement new component which manages global configuration of the automation framework

Rationale: Currently we have multiple configuration files for different components. For administrator it's a bit troublesome to modify them all instead of having a single global configuration file.

Decision: implement. Effort estimate: Small.

[150506-6 a4virtan] Implement new component which collects additional information from the target before execution begins

Rationale: We should be able to collect a lot of useful information from target's LDAP which is useful to be populated to different variables to be available to testcases.

Decision: implement little bit later. Effort Estimate: Moderate

[150506-7 a4virtan] Move sourcecodes from CVS to SVN repository

Rationale: CVS is missing good support for renaming and moving files.

Decision: implement. Effort Estimate: Moderate

[150506-8 a4virtan] Use Nokia naming convention for packages

Rationale: We should use standard naming conventions. Also sourcecode directories should follow standard naming convention since some tools are expecting that (Javadoc, JTest, Eclipse): e.g. /com/Nokia/...

Decision: implement. Effort Estimate: Moderate

[160506-1 a4virtan] Use Java 1.5 for some components which doesn't require 1.4 compatibility

Rationale: Provides generics which would make current code more typesafe since we're using a lot of containers instead of native arrays.

Decision: Java 1.5 will be used where it must be in the first phase.(on case bases)

[160506-2 a4virtan] Start to use Maven as a build system

Rationale: Maven provides standard way to build java projects. Currently we use customized Ant build scripts which are not propably so extensible or easy to understand.

Decision: implement little later. Effort Estimate: Moderate

[050606-1 a4virtan] Improve database structure

Rationale:

  • We are storing application heartbeats to the same database where also testcase results are. These should be separated.
  • Testcase/domain/release relations are not done properly. Currently testcases belong to the release and domain is extra information for the testcase. Instead testcases should belong to the domain (which can be recursive) and domains belong to the release. (Not critical problem. Would only make it easier to understand database structure).
  • Host/Node/Cluster relations are broken. Uniqueness of the Host is invalid.
  • Currently we are only storing the pass/fail result of the testcase although results can be much more complex. This was valid one year ago when tcexec was only way to execute testcases. Currently testcase can consists also many tasks which prepare environment for the testcase and those tasks can also fail. Instead of trying to store all the information to the single table we should have some table hierarchy for the results.
  • Database should consists of mandatory and optional tables. For example user profiles for the web pages could be part of the same database but the tables would not be mandatory. Database should be constructed by some Manager component. For example UpdateManager-component when started first time could construct the initial database unless database is already available.

[050606-2 a4virtan] Implement new component which manages plugins

Rationale: Since we have multiple extension points for plugins in the framework we should have single way to control them. For example DataManager-component could manage all the plugins for other components.

[060606-1 a4virtan] Implement proper file streaming over RMI

Rationale: Currently Scheduler expects to see same directories as the user. This doesn't work too well if user uses AFCli from some remote host which doesn't export the user's directory. So all results which are written outside /twafs should be streamed back to the AFCli. Also testcases executed with --path should tranfer the files to the scheduler.

[120606-1 a4virtan] Refactor domain/subdomain/subsystem concept

Rationaile: Currently we have hardcoded domain and subdomain. Instead we should have recursive domains. Also subsystem-metadata should be renamed to domain-metadata. Subsystem-metadata is now also only parsed from subdomain-level. Instead all domains could have own metadata which is then inherited to child domains from parents.

Changes are needed to existing metadata files (XMLConverter should be used to process them), to the database (DBUpdate-tool should be used for that) and to some components (AFCli, Scheduler, MetadataParser, ExecutionParser, JobPlanner) althought required changes to them are simple to implement. Changes should be done after DataManager-component is ready. The biggest effort is to update old webpages, since big part of existing SQL queries will break when we change the database structure.
