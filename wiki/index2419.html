Sun's Java coding conventions are used with the following additions.

Stick to one coding convention in single project. If you feel that it should be changed, then change it to whole project.

[edit] Indentation

  • 4 spaces used for indentation, no tabs please. Correct bracket handling: 
      try
      {
          foo();
      }

or 

     try {
         foo();
     }
  • Use empty line between functions 
     class Foo
     {
         void func1()
         {
             ...
         }

         void func2()
         {
             ...
         }
     }
  • If using emacs/xemacs, add following to your init.el: 
     (setq load-path (cons (expand-file-name "/twafs/utilities/emacs") load-path))
     (require 'flexitaf-mode)

That file is mostly copy-paste from: http://portals.apache.org/development/code-standards.html. Additionally it uses java-mode-indent-annotiations.el to fix java annotiation indentation rules.

  • To ease up reformatting existing files, this helps a lot (but propably only works with Xemacs): 
     (defalias 'indent-all
       (read-kbd-macro "C-x h ESC C-\\"))
     (global-set-key 'f12 'indent-all)

So whenever you press F12, your current file will be reformatted.

[edit] Ordering

  • Use following defined in Java class file content ordering with following addition: static classes should be declared between 3-4.

[edit] Naming

  • Interface names start with capital 'I' 
   public interface IFoo
   {
       void myFunction(Argument arg) throws Exception;
   }
  • Member variables start with 'm_'
  • Static variables start with 's_'
  • Static final variables are written in capital letters (underscores '_' separate words)
  • Local variables don't use prefixes
  • Use descriptive variable names 
   public class Foo implements IFoo
   {
       public static final int FOO_CODE = 1;
       private int m_fooState;
       private static Foo s_singletonInstance;

       public void myFunction(Argument arg) throws Exception
       {
           Bar bar = new Bar(arg);
           int i = 0;
       }
   }

[edit] Documenting

  • Write comments using Javadoc style. [1]
  • Use only English when writing comments.
  • Don't document self describing code since it only makes it harder to read. (This rule should be removed when we start to use IDE with an editor which can hide javadocs)
  • Tip: In Emacs you can use fill-paragraph command (M-q) to easily wrap long lines also inside a comment block. 
   public class User
   {        
       ...
       
       /**
        * Returns the name of the user
        *
        * @return the name of the user
        */
       public String getUserName()
       {
           return m_user; //return name of the user
       }
   }
  • Document also parts of the implementation which might be very hard to understand 
   public class Foo
   {        
       ...
      
       public String doSomething() throws FooException
       {
           try
           {
               SomeLib.Bar.doIt();
           }
           catch(SomeLibException e)
           {
               e.setSomeObj(null); //obj might not be serializable and FooException must be fully serializable
               throw new FooException(e);
           }
           
           // This comment concerns the following two lines
           doSomethingWorthCommenting();
           doSomethingMoreWorthCommenting();
           
           notWorthCommenting();
       }
   }

[edit] Coding

  • Follow rules defined in Joshua Bloch's Effective Java Programming Language Guide

[edit] Testing

  • Place unit tests to the same package but to the different hierarchy. Otherwise you cannot test package scoped code (which is very useful when you want to create methods to ease up testing, like special constructors, but you don't want them to be part of public API): 
   src/
     main/
       java/
         foo/
           bar/
             MyClass.java
   src/
     test/
       java/
         foo/
           bar/
             MyClassTest.java

[edit] Property and config parameter naming

  • Property names: 
   com.nokia.flexi.flexiaf.<component>.globalProperty
   com.nokia.flexi.flexiaf.<component>.[<package>].<class>.privateProperty
  • Config parameter names: 
   <flexitaf>
     <configuration>
       <component-name>
         <component-property>bar</component-property>
       </component-name>
     </configuration>
   <flexitaf>

[edit] Resources

  • Store resources to class hierarchy. Under class hierarchy you can use whatever structure you want. 
   src/
     main/
       java/
         foo/
           bar/
             MyClass.java
   src/
     main/
       resources/
         foo/
           bar/
             MyClass.properties
             properties/
               foo.properties
             config/
               config.xml

[edit] Logging

  • System.out or System.err should never be used directory.
  • java.util.logging is preferred way to do logging. com.nokia.flexi.flexitaf.util.logging provides extension for java.util.logging to enable publisher-subscriber oriented logging, which is useful in client-server messaging.
  • Libraries should always use logging level < INFO.
  • In some special cases libraries can use WARNING or SEVERE levels if it is impossible to propagate some error to the application.
  • Applications should use INFO for normal user-visible logging.
  • Avoid hard-coded trace logging. Use aspects instead to weave your classes.
  • Access logger through following package private Utils class: 
package com.nokia.flexi.flexitaf.repository;

import java.util.logging.*;
 
class Utils
{
    private static Logger s_logger = Logger.getLogger(Utils.class.getPackage().getName());

    private Utils() { }

    public static Logger getLogger()
    {
        return s_logger;
    }
}

[edit] Exceptions

  • Use org.apache.commons.lang.NullArgumentException for errors when you detect null argument.
  • Use IllegalArgumentException for errors when otherwise invalid argument has been detected.
  • Use UnsupportedOperationException to mark code blocks which are not yet implemented.
  • Use IllegalStateException for cases when object/objects are in such a state that certain operation cannot be done or is not expected.
  • Avoid using NullPointerException. This is normal indication of coding errors detected by JVM and makes error analysis harder when same exception type is used intentionally in code.

[edit] Configfile templates

  • Parameters which are required by default should have reasonable default value
  • Parameters which aren't required by default should be commented out
  • All parameters should have commented description
  • All description should contain information about used default value
  • Example: 
<settings>
 <!--
  | The path to the local repository maven will use to store artifacts.
  |
  | Default: ~/.m2/repository
 -->
 <localRepository>/localhome/a4virtan/.m2/repository</localRepository>
 <!-- interactiveMode
  | This will determine whether maven prompts you when it needs input. If set to false,
  | maven will use a sensible default value, perhaps based on some other setting, for
  | the parameter in question.
  |
  | Default: true
 <interactiveMode>true</interactiveMode> 
-->
</settings>

[edit] Optimization

  • Use correct container type General information about collections Performance comparasion of ArrayList and LinkedList
    • ArrayList is faster than LinkedList for random access and for algorithms that follow a random access pattern, such as binary searching.
    • LinkedList is faster than ArrayList for repeated insertions to the front of the list.
    • HashMap lookup is faster than binary search on an ArrayList.
    • Appending elements to the end of a list has a fixed averaged cost for both ArrayList and LinkedList. For ArrayList, appending typically involves setting an internal array location to the element reference, but occasionally results in the array being reallocated. For LinkedList, the cost is uniform and involves allocating an internal Entry object.
    • Inserting or deleting elements in the middle of an ArrayList implies that the rest of the list must be moved. Inserting or deleting elements in the middle of a LinkedList has fixed cost.
    • A LinkedList does not support efficient random access.
    • An ArrayList has space overhead in the form of reserve capacity at the end of the list. A LinkedList has significant space overhead per element.
  • When concatating string, use StringBuilder 
   String foobar = "foo" + "bar" + "foo" + "bar"; //wrong
   String foobar = new StringBuilder("foo").append("bar").append("foo").append("bar").toString(); //correct
  • Use primitive types rather than corresponding objects unless you really have need for null-value. 
   public Integer appendValues(Integer arg1, Integer arg2) //wrong
   public int appendValues(int arg1, int arg2) //correct
  • For complex logging operations verify first would the message even be logged 
   logger.fine(doSomeComplexOperation()); //wrong
   if(logger.isLoggable(Level.FINE)) logger.fine(doSomeComplexOperation()); //correct
    • Don't do it for simple logging operations! Otherwise the checking will be done twice! 
   is(logger.isLoggable(Level.FINE)) logger.fine("foo" + "bar"); //wrong
