[edit] Introduction

This is a hands-on tutorial on using FlexiTAF to test FlexiPlatform in Espoo/Karaportti environment.

  • FlexiTAF tool home page with brief introduction. (NSN Wiki)
  • User-guides for the latest released version (pdf).
  • PD Test Automation Portal (NSN Wiki)
  • Training material
    • Using FlexiTAF and Robot Framework to test FlexiPlatform (FP5 Cat scope, ppt)
    • Test Automation in FlexiPlatform - Demo and Hands-on Training. (FP4 Charm scope, ppt)
  • All FlexiTAF documents in Sharenet-IMS (doc & ppt).
  • The latest version of this Wiki-page can be always found from http://sugar.fp.nsn-rdnet.net/wiki/index.php/FlexiTAF_Tutorial.

[edit] How to access

  • You need an user account for the FlexiPlatform Espoo Lab NIS. NIS (Network Information Service) is the local user database in the Espoo Lab. Regarding the NIS user accounts, please contact FlexiPlatform Espoo Lab support either by:
    • Creating a TSJIRA issue through the Web tool http://tsjira.emea.nsn-net.net:8080/browse/LAB , OR
    • sending an email to the TSJIRA’s email interface espoo.platform_labservices@nsn.com


[edit] Afcli (CLI-Interface)

  • There are currently three maintained client hosts where you can execute and develop testcases
    • Flexiaf1.fp.nsn-rdnet.net
    • Flexiaf2.fp.nsn-rdnet.net
    • Flexiaf7.fp.nsn-rdnet.net
  • Note that it is also possible to install client to any other host (Windows or Linux) which has Java JRE5 or better installed. Installation package can be found from here.

[edit] WebUI

  • Web-interface is running at http://sugar.fp.nsn-rdnet.net/flexitaf

[edit] Testcases

  • Testcases are stored to SVN-repository which is administrated by Tools-project. Use following URL to access it: http://sugar.fp.nsn-rdnet.net/svn/testcases.
  • For more information about how to access testcases check VersionControl FAQ.

[edit] Results

  • Results from the real executions are stored to /twafs/results which is mounted to all flexiaf-hosts.
  • Results from local executions (when you use --local-option with CLI) are stored to the placed defined in user's config-file ($HOME/.flexitaf/flexitaf.xml). For more information about how to create initial config-file see Initial setup instructions.

[edit] Targets

  • FlexiTAF maintains a list of known targets (SUTs, Systems Under Test) in a database. Targets are described using a XML-based language. For FlexiTAF to be able to fetch necessary information of the target before executing the testcase you need to have /TargetInfo recovery group deployed and unlocked in the FP cluster.
  • There is separate Target Inspector tool available for examining the SUTs.

[edit] Initial setup

  • When you setup Afcli for the first time you need to create configuration file to your $HOME directory: 
mkdir $HOME/.flexitaf
vi $HOME/.flexitaf/flexitaf.xml
  • The content of this configuration file should be following: 
<flexitaf>
  <configuration>
    <testcase-repository> 
      <releases>
        <release name="4.charm" path="${user.home}/testcases/rel_4.charm" result-path="${user.home}/results/rel_4.charm"/>
      </releases>
    </testcase-repository>
    <afcli>
      <targets>
        <target>
          <name>isengard</name>
          <username>root</username>
          <password>root</password>
        </target>
        <target>
          <name>Rivendell</name>
          <username>root</username>
          <password>rivendell</password>
        </target>
      </targets>
      <properties>
        <property>
          <name>flexitaf.task.tcexec.log-level</name>
          <value>1</value>
        </property>
        <property>
          <name>flexitaf.task.tcexec.verbose</name>
          <value>false</value>
        </property>
      </properties>
    </afcli>
  </configuration>
</flexitaf>
  • <testcase-repository> element is used to specify location of locally checked out testcases.
  • <targets> element is used to associate credentials for the clusters you want to use. This saves you from having to add --user and --password in CLI everytime you run the testcase.
  • <properties> elements is used to define default values to metadata-attributes. More information about these can be found from user-guide.

[edit] Basic commands

  • Afcli uses following structure for all the commands 
afcli <OPTIONS> <COMMAND> <COMMAND ARGUMENTS>
  • There are set of global options which affect all the commands (like setting the verbosity-level) and command specific options.

[edit] Show help 

afcli
afcli help exec

[edit] Control verbosity 

afcli -d 3 list releases
  • Verbosity levels are following
    • 0 = INFO
    • 1 = FINE
    • 2 = FINER
    • 3 = FINEST

[edit] List content of repository

  • List releases 
afcli list releases
  • List FunctionalTesting subdomains of release 4.charm 
afcli -r 4.charm list domains FunctionalTesting
  • List Testcases of domain FucntionalTesting/HighAvailabilityServies/SS_CFM of 4.charm 
afcli -r 4.charm list testcases FunctionalTesting/HighAvailabilityServices/SS_CFM
  • Local repository content can be accessed by adding --local-option 
afcli --local list releases

[edit] Show testcase details 

afcli -r 4.charm show testcase FunctionalTesting/HighAvailabilityServices/SS_CFM FT_HAS_CFM_001

[edit] Create new testcase 

afcli -r 4.charm --exec-engine tcexec create testcase Sandbox/Testing TEST_001
  • NOTE: If the domain-directory does not exist, it must be create manually due to existing bug. [1]
  • See also Instructions for writing testcase scripts and Test Automation related topics

[edit] Manage targets

  • List available targets 
afcli list targets
  • Show target Moria 
afcli show target Moria
afcli -d 3 show target Moria
  • Add new target TestMoria. For more information how to prepare cluster for this check Instruction for enabling /TargetInfo 
afcli add target TestMoria tcp://10.20.92.186:443
  • Modify target TestMoria by setting new TargetInfo URI. 
afcli modify target TestMoria uri=tcp://10.20.92.176:443
  • Delete target TestMoria 
afcli delete target TestMoria
  • If TargetInfo is not available it is possible to write topology-description manually 
afcli -f topology.xml add target TestMoria

[edit] Manage testcase executions

  • To cancel job at any time press CTRL-C
  • To detach job at any time press CTRL-D
  • Run testcase TEST_ANALYZE_001 from 4.charm in Moria using credentials root/moria 
afcli -r 4.charm -t Moria -u root -p moria exec TEST_ANALYZE_001
afcli -r 4.charm -t Moria -u root -p moria -d 3 exec TEST_ANALYZE_001
  • Run testcase TEST_ANALYZE_001 from 4.charm in Moria using credentials root/moria and automatically detach job 
afcli -r 4.charm -t Moria -u root -p moria --detach exec TEST_ANALYZE_001
  • Run three instances of testcase TEST_ANALYZE_001 in parallel from 4.charm in Moria using credentials root/moria 
afcli -r 4.charm -t Moria -u root -p moria exec '(TEST_ANALYZE_001,TEST_ANALYZE_001,TEST_ANALYZE_001)'
  • Run all testcases which match to regex-pattern TEST.* from 4.charm in Moria using credentials root/moria 
afcli -r 4.charm -t Moria -u root -p moria exec 'TEST_.*@regex'
  • List scheduled jobs 
afcli list jobs
  • Show details of specific job 
afcli show job <JOB_ID>
  • Attach specific job 
afcli attach job <JOB_ID>
  • Cancel specific job 
afcli cancel job <JOB_ID>
  • Delete specific job 
afcli delete job <JOB_ID>
  • To execute testcases from local-repository during development you need to use --local-option 
afcli --local -r 4.charm -t Moria -u root -p moria exec TEST_ANALYZE_001
  • Note that from local-executions results aren't stored to anywhere else than your own result-directory. Those won't be visible either from WebUI.
  • Only the author of the job (and administrator) can access other persons jobs.

[edit] Debugging

  • When you encounter some problem you can get more output (and stacktraces) with -d 3-option.
  • When you are unsure how exec-command forms the execution you can debug it using --dump-execution-command.
  • You can find known bugs from issue-tracker

[edit] Metadata

  • XML-based language (Metadata Language) is used to describe testcases and domains. Testcases will always inherit definitions from their parent-domain metadata.
  • Metadata can be used to describe
    • How to prepare testcase for execution. The testcase script files can contain references to variables which are then substituted by real values during preparation.
    • How to execute the testcase. Which execution tool should be used.
    • What resources testcase requires.
    • What are constraints for the environment where testcase is executable.

[edit] Metadata processing

  • Metadatas are processed in following phases
  1. Pre
  2. Execute
  3. Analyze
  4. Post
  5. Report
  • For each phase we can bind tasks to be executed.

[edit] Writing testcase scripts

  • Primary engine for executing testcases in FlexiTAF is TCExec.

[edit] TCExec

  • TCExec operates on files placed in tcexec directory in the testcase directory eg. Sandbox/Testing/TEST_001/tcexec
  • File named tc contains the testcase logic.
  • The control of the testcase execution is in FlexiTAF server ouside the target cluster. %-macros are used to open connections to the target and other hosts. Each %-macro starts a new script.
  • There are macros for ...
    • executing commands in target node (%NODE)
    • in other hosts (%HOST)
    • in local host (%LOCAL)
    • in background (%BACKGROUND NODE, %BACKGROUND HOST)
    • simple control flow (%SLEEP, %BACKGROUND KILL).
  • There is always implicit %NODE macro inserted in the beginning of the tc script.
  • There are environment variables available:
    • Testcase exit codes: TCE_EXIT_PASS, TCE_EXIT_FAIL, TCE_EXIT_ERROR, and TCE_EXIT_WARNING
    • Script home: TCE_HOME
    • Place for results: TCE_RESULT_DIR
    • In analyze script, standard output of foreground and background processes: TC_OUT, TC_OUT_BG_id
  • tc_analyze file contains the logic for analyzing if the testcase passed of failed. (optional)
  • Connecting the target with TCExec very simple. This is a complete tc-script for testing Directory swithcover in FP4 Charm: 
%NODE none
#!bin/bash
uname -n
fshascli -w -F -n /Directory

%NODE none
#!bin/bash
uname -n
  • For more examples, see TCExec Examples
