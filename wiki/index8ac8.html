General Information

This wiki page tries to consolidate the common issues encountered by FlexiTAF users and their solutions. This page also lists the known bugs and their possible work-arounds.

This wiki page and the FlexiTAF user guides are the first points of reference when you encounter any issues / queries on FlexiTAF. If these documentations do not have sufficient solution for your problem, please mail your query to flexitaf-support@internal.nsn.com mailing list. Please attach all the relevant information while sending the mails. (When you are collecting the output of afcli commands, please use "-d 3 --debug" option with the command to increase the log-level)

Testcase & JobPlan Execution

Testcase Not Found

  • Changes to the testcase repository will not be reflected immediately after the commit. It may take time. Please be patient for atleast 2 hours, before you approach FlexiTAF Support
  • Make sure that the follwing rules are obeyed while deciding the directory structure & names
    • Testcase names should match with the pattern ^[0-9A-Za-z_-]*_[0-9]{3,}$
    • Directories with any other naming pattern is considered as domain/subdomain.(Exceptions:Directories with the name "common" and its sub-directories are not considered as domains, they are used to store common files)
    • Testcase & domain names are not allowed to contain any special characters, except hyphen(-) and underscore(_)
    • Testcases cannot be placed directly under the release directory (eg. rel_5.lynx,rel_6.cougar).There should be atleast one domain level.
    • It is not allowe to have both sub-domains & testcases together under the same domain/subdomain
  • For local executions, you need to create configuration file to your $HOME directory & configure it correctly 
mkdir $HOME/.flexitaf
vi $HOME/.flexitaf/flexitaf.xml

Below is a sample template of the configuration file 

<flexitaf>
  <configuration>
    <testcase-repository> 
      <releases>
        <release name="4.charm" path="${user.home}/testcases/rel_4.charm" result-path="${user.home}/results/rel_4.charm"/>
        <release name="5.lynx" path="${user.home}/testcases/rel_5.lynx" result-path="${user.home}/results/rel_5.lynx"/>
      </releases>
    </testcase-repository>
    <afcli>
      <targets>
        <target>
          <name>isengard</name>
          <username>root</username>
          <password>root</password>
        </target>
        <target>
          <name>Rivendell</name>
          <username>root</username>
          <password>rivendell</password>
        </target>
      </targets>
      <properties>
        <property>
          <name>flexitaf.task.tcexec.log-level</name>
          <value>1</value>
        </property>
        <property>
          <name>flexitaf.task.tcexec.verbose</name>
          <value>false</value>
        </property>
      </properties>
    </afcli>
  </configuration>
</flexitaf>

<testcase-repository> element is used to specify location of locally checked out testcases.release path should be correctly pointing to the location where the testcases from the corresponding release are stored. <targets> element is used to associate credentials for the clusters you want to use. This saves you from having to add --user and --password in CLI everytime you run the testcase. <properties> elements is used to define default values to metadata-attributes. More information about these can be found from user-guide.

Warning: Ensure that the result path "result-path" and the testcase path "path" are not the same and are not overlapping, as it might result in execution problems


Failed to Get Testcase Metadata - Caused by Parse Error

  • Make sure that the testcase.xml & all the parent domain.xmls are having proper xml syntax
  • Make sure that the metadata language used in the above mentioned xmls are correct. More info about the metadata language can be found in the FlexiTAF User guide, Metadata Language chapter


Connection cannot be established to <target-ip>

  • Make sure that the target is up & running , and is accessible over SSH


Domain has wrong type of metadata or Testcase has wrong type of metadata

  • domain.xml is starting with <testcase> tag. Correct it to <domain> tag, OR,
  • testcase.xml is starting with <domain> tag. Correct it to <testcase> tag.


Parsing '/opt/flexitaf-server-<version>/work/job-cache/job-<id>/temp/jod<id>temp/<file>' failed: Data source does not exist

  • Please check the name of the testcase directory & other testcase files in testcase.xml/domain.xml, and make sure that the directory/files mentioned exist.

Cannot write: No space left on device

  • Please check testcase/domain.xml, $HOME/.flexitaf/flexitaf.xml. flexitaf.task.tcexec.target-cleanup will be false. Change the value to true to enable the target clean-up.

Testcase Repository Management

Quality Center Reporting

Test case not found from the Quality Center under the path <release-path>

  • Make sure that a testcase with the same Testcase ID as FlexiTAF-Testcase-Name, is existing in QC TestPlan under the path (either directly under this path or under its sub-folders) mentioned in the error message. Note that these matchings are case-sensitive.

Found multiple test case ids for the given test case

  • There are more than one testcase with the same ID in QC TestPlan, under the path /Subject/FlexiPlatform/<release>. This is not allowed for FlexiTAF->QC reporting. Delete the duplicate ones.(You can search the duplicates using the testcase names mentioned in the error message)

Illegal character entity: expansion character not a valid XML character

  • This error occurs when the test case description in QC contains control characters.Those control chars break XML parsing on FlexiTAF side because they are not valid XML.As a workaround you can try to identify problematic test case from the QC and remove possible control characters from its description.

Read timed out or Invalid server response

  • “Read Time out” error may happen when there is some network issue / QC is inaccessible etc. This happens very occasionally. If you try reporting after sometime, it generally works.
  • If the above solution doesn't work: If the testset, to which the runs are being reported, has large number of testcase instances, FlexiTAF->QC will take longer time for querying the testset, and it may get timed out also if the testset is extremely large. According to QC-Spport,it is not recommended to use very large number of testcases in a single testset.Please break the testset into multiple testsets, for a better & reliable performance. (This issue is in discussion with QC-Support. Till it get resolved, larger testsets cannot be used for reporting to, please break the testset into multiple smaller testsets)

Server not available

  • “Server not available” error generally happens when there is some network issue / QC is inaccessible etc. This happens very occasionally. If you try reporting after sometime, it generally works.

Node not found or TestSet not found from the Quality Center

  • Make sure that the Test Set you are trying to report to, is existing in QC TestLab. Take extra care about the spelling, white spaces etc

QcXml.Exceptions.ExceptionConnectionFailed: Wrong user password

  • Mail to FlexiTAF Support

Reporting gets stuck at "Resolving Quality Center tests in test set"

  • If the testset, to which the runs are being reported, has very large number of testcase instances, FlexiTAF->QC will take longer time for querying the testset. According to QC-Spport,it is not recommended to use very large number of testcases in a single testset.Please break the testset into multiple testsets, for a better & reliable performance.

Build Provider

Unknown component interface: interface com.nokia.flexi.flexitaf.buildprovider.client.IBuildProviderClientComponent

  • Mail to FlexiTAF Support

Compatible build provider not found

  • Check the build on the target. If it is not a flexman build or CI build, there is high chance that FlexiTAF cannot recognize it.In such cases where you use your own compiled test builds, atleast make sure that it follows the Flexman build naming pattern.

Error while resolving binary cache path - Caused by: File <somepath>/xpath does not exist

  • FlexiTAF caches the generated binaries for future use.The above error occurs when the cached binary is corrupted.You need to force recompilation of the binary once to avoid this error.You can force recompilation on commandline by using "–D flexitaf.task.build.force-recompile=true" along with the afcli execution command.(Note:You need to force recompilation only for the next immediate execution, not for all the future executions)
