The main metadata concepts that are used for the FP5-Cat automated FT testcases is described here. The testcase metadata are used for setting up the regression suites by using the FlexiTAF jobplan concept. The FP5-Cat jobplans are described here.

HOWTO Introduction

The metadata settings can be done in the testcase metadata file (testcase.xml) or in the domain metadata (domain.xml). The effective metadata of a testcase are the combination of its domain, subdomain and testcase metadata. The inheritance of metadata is described in the FlexiTAF User Guide, section 4.1, as follows:

Metadatas are XML-files which are placed in the testcase repository either in domain directories (to be interpreted as domain metadata) or in testcase directories (to be interpreted as testcase metadata). Thereby, each domain and each testcase can have its own metadata file. Domain-level and testcase-level metadata files can describe the same type of information and use the same syntax. During testcase execution, the effective metadata is obtained from (1) the testcase metadata file, (2) the testcase’s domain metadata file, (3) the domain’s domain metadata file, etc. Thereby, each metadata file inherits from its parent’s metadata file using the following rules:
  • Metadatas always override values defined in parent metadatas (e.g., testcase metadatas override domain metadatas).
  • Lists of subelements are combined. For example, the lists of pre tasks defined on all levels (e.g., domain-level, sub-domain level and testcase-level) are combined in the effective metadatas.

The effective metadata of a testcase can be checked as follows: 

 afcli show testcase -r 5.cat FunctionalTesting/DOMAIN TESTCASE_ID

testcase.xml 

 <testcase model-version="1.0.0" description="">
     <constraints>
         <!-- constraints are used to determine on which HW types the testcase is applicable and also to determine other target characteristics needs -->
     </constraints>
     <properties>
         <!-- properties are used to determine the automation level of a testcase, the type of testcase and its status -->
     </properties>
     <variables>
         <!-- normal testcase variables -->
     </variables>
 </testcase>

domain.xml 

 <domain model-version="1.0.0" description="">
     <constraints>
         <!-- constraints are used to determine on which HW types all testcases in the domain are applicable and also to determine other target characteristics needs -->
     </constraints>
     <properties>
         <!-- properties are used to determine the automation level of all testcases in the domain, their type of testcase and their status -->
     </properties>
     <variables>
         <!-- normal domain-level variables -->
     </variables>
 </domain>

Determining on which HW types a testcase can be run

With constraints, it is possible to determine if a testcase can be executed on that type of HW or not. The names of the HW types are those known by TargetInfo service which just uses the cluster's variable "HW_PLATFORM".

  • HW types in FP4-Charm
    • NokiaBlade
    • IBMBladeCenter
    • HPBladeSystem
    • HPProLiant
    • MCP18
  • HW types in FP5-Cat:
    • AdvantechFWA
    • ATCA
    • VMware
  • HW types in FP5-Lynx:
    • AdvantechFWA
    • ATCA
    • VMware
    • BCN
    • ...

Determining that it can be run on all HW types

No constraints with any HW type names are in the testcase's effective metadata. 

 <constraints>
   <!-- nothing in here using flexitaf.target.hw -->
 </constraints>

Determining that it can only be run on certain HW types

Following constraint determines that the testcase with this metadata can only be executed on targets of type "HW_type_1" and "HW_type_2" and will exit with ERROR if scheduled for execution on any other HW type. 

 <constraints>
     <or>
         <equal arg1="${flexitaf.target.hw}" arg2="HW_type_1" />
         <equal arg1="${flexitaf.target.hw}" arg2="HW_type_2" />
     </or>
 </constraints>

Determining that it cannot be run on certain HW types

Following constraint determines that the testcase with this metadata can not be executed on targets of type "HW_type_1" and "HW_type_2" and will exit with ERROR if scheduled for execution on them. This also means that it can be executed on all other HW types.
Previous variables "FT_supportedhw" and "supportedhw" have been converted into this kind of metadata. 

 <constraints>
     <not> <equal arg1="${flexitaf.target.hw}" arg2="HW_type_1" /> </not>
     <not> <equal arg1="${flexitaf.target.hw}" arg2="HW_type_2" /> </not>
 </constraints>

Transforming old metadata variables "FT_supportedhw" and "supportedhw" into the new concept

Old metadata variables "FT_supportedhw" and "supportedhw" have been transformed into constraints as follows:

  1. A list of supported HWs has been transformed into a list of the respective unsupported HW types (all using just the Charm HW information)
  2. If the variable was set to generic, no constraint has been defined.

Example
old way 

 <testcase model-version="1.0.0" description="">
     <variable name="FT_supportedhw" value="NokiaBlade,IBMBladeCenter,HPProLiant"/>
     <tasks>
         <execute>
             <tcexec timeout="300"/>
         </execute>
     </tasks>
 </testcase>

new way 

 <testcase model-version="1.0.0" description="OS: CPU affinity">
     <constraints>
         <not>
             <equal arg1="${flexitaf.target.hw}" arg2="HPBladeSystem" />
         </not>
         <not>
             <equal arg1="${flexitaf.target.hw}" arg2="MCP18" />
         </not>
     </constraints>
     <tasks>
         <execute>
             <tcexec timeout="300"/>
         </execute>
     </tasks>
 </testcase>

Script for changing supportedHW into "not supported HW constraints"

I have done the changes with a script which you can find here (use on your own risk!): flexiaf1:/twafs/utilities/MetadataConversion/change_supportedHW2constraint.sh The easiest way to do it:

  1. copy the testcases and domain.xml files from Charm (see VersionControl_FAQ_-_Copying_testcases_from_old_release_to_new_one)
  2. commit the changes (message: "copied from Charm")
  3. in your domain directory do: find . -name "*.xml" -exec /twafs/utilities/MetadataConversion/change_supportedHW2constraint.sh {} \;
  4. check the changes and commit

Determining PI testcases

Two types of testcases are distinguished: PI testcases and "normal testcases". PI testcases are those which are listed in the testplans in the "Product Integration Testcases" section. The "normal testcases" are all the others (i.e., testcases taken from previous release as well as new testcases).

The type of testcase is determined by the property "ProductIntegrationTestcase" which has to be set to "true" if it is a PI testcase. For normal testcsaes, it has to be empty/unset or set to "false".

If a testcase has the following effective metadata, it is determined to be a PI testcase. 

 <properties>
     <property name="ProductIntegrationTestcase" value="true" />
 </properties>

Determining the automation level

Three different automation levels are distinguished:

  • fully automated: the testcase does not need any manual steps in the execution or analysis; therefore, it can be included in regression suites
  • partly automated: either test execution or test analysis needs some manual step(s); the testcase is available in the testcase repository to support manual (regression) testing
  • manual: used when the testcse repository is used as a storage place for helper files which are needed in manual testing

The automation level is determined by setting the property "AutomationLevel" to one of the three allowed values "fullyAutomated", "partlyAutomated" or "manual". Since by default all testcases are set to be fully automated, it has only to be changed if the default value is not correct. 

 <properties>
     <property name="AutomationLevel" value="partlyAutomated" />
 </properties>

Determining that status of a testcase

Two intermediate statuses are distinguished which can be set to avoid the testcase's execution in certain regression suites:

  • testcase is "under investigation": test which is not passing and the problem has been reported to the testcase bug database for investigation; while this is pending, it will not be included in the normal regression suite
  • testcase detects a known Pronto: testcase which is not passing because of a reported problem (pronto) in the SUT

Status "OpenBug"

The internal (not further distinguished) substates of this status are:

  1. testcase bug reported, but not yet analyzed
  2. analysis ongoing

The status is removed from the testcase's metadata if

  • the analysis has revealed a pronto that has been reported (see status "OpenPronto") or
  • the problem of the testcase has been fixed

The status "under investigation" is determined by the property "OpenBug" set to the (list of) bug IDs (e.g., #1234,#2345). 

 <properties>
     <property name="OpenBug" value="#1234,#2345" />
 </properties>


Status "OpenPronto"

The internal (not further distinguished) substates of this status are:

  1. pronto reported and open
  2. pronto ready for testing

The status is removed from the testcase's metadata if

  • the pronto is closed (after testing)
  • the pronto has been set to CNN

The status "detects a known Pronto" is determined by the property "OpenPronto" set to the (list of) Pronto IDs (e.g., 12345ESOE01, 23456ESPE01). 

 <properties>
     <property name="OpenPronto" value="12345ESOE01,23456ESPE01" />
 </properties>
